# 第4章：情報を自分で集める力を身につける

**目安時間：4時間**

> **前提：** 自分のトラックの第3章のセットアップが完了していること（CLIツールが使える状態）

> AIツールは進化が速い。今日正しい情報が3ヶ月後には古くなっていることもある。このコンテンツを読み終えた後も自走できるように、「どこで調べるか」「どう聞くか」を身につける章。

---

## この章で学ぶこと

- 公式ドキュメントを読むべき理由と読み方を理解する
- 自分が使うAIツールの公式ドキュメントの構造を把握する
- AIを使って情報収集する方法を身につける
- 情報の鮮度と信頼性を判断できるようになる
- 「わからないこと」をAIに正確に伝える技術を得る

## 4-1. 公式ドキュメントはなぜ読むべきか

### ブログ・Qiita・記事との違い

ネットで調べると、ブログや技術記事がたくさん出てくる。読みやすいし、すぐ使えそうな情報が多い。でも、これらには共通の問題がある。

**書かれた時点の情報しか保証されない。**

AIツールのバージョンが変わったり、APIの仕様が変わったりすると、記事の内容が古くなる。古い情報をそのまま試してエラーが出て詰まる、というのはよくある話。**AIツールは特に進化が速く、半年前の記事が既に古いことも珍しくない。**

一方、公式ドキュメントは開発元が管理しているため：

- **最新の仕様が書かれている**
- **動作が保証されている手順が書かれている**
- **変更があった場合に更新される**

公式ドキュメントを読む習慣がある人とない人では、詰まったときの解決速度が大きく変わる。

### 「難しそう」という先入観を捨てる

公式ドキュメントは英語が多い、難しい言葉が並んでいる、と感じる人は多い。でも全部読む必要はない。

**目的のページに直接飛んで、必要な部分だけ読む** で十分。わからない単語はAIに説明してもらえばいい。英語のページなら丸ごとAIに渡して要約させてもいい。

## 4-2. 自分が使うツールの公式ドキュメントの構造と読み方

### 主要なドキュメントの場所

**Gemini トラック：**

| ドキュメント | URL | 対象 |
|---|---|---|
| Google AI for Developers | [https://ai.google.dev/](https://ai.google.dev/) | API・モデル・開発者向け |
| Gemini for Google Workspace | [https://workspace.google.com/products/gemini/](https://workspace.google.com/products/gemini/) | 企業向けサービス・データポリシー |
| Gemini CLI | [https://github.com/google-gemini/gemini-cli](https://github.com/google-gemini/gemini-cli) | CLI ツールの使い方 |

**Claude トラック：**

| ドキュメント | URL | 対象 |
|---|---|---|
| Claude Code ドキュメント | [https://code.claude.com/docs](https://code.claude.com/docs) | Claude Code の使い方・設定 |
| Anthropic ドキュメント | [https://platform.claude.com/docs](https://platform.claude.com/docs) | モデル・API・開発者向け |
| Anthropic Privacy Center | [https://privacy.anthropic.com/](https://privacy.anthropic.com/) | データポリシー |

### 読み方のコツ

**①「Getting Started（はじめに）」から読まない**

目次を見て、今知りたいことが書いてあるセクションに直接飛ぶ。

**② ページ内検索を使う**

`Ctrl + F`（macOS は `Command + F`）でページ内検索して、知りたいキーワードを探す。

**③ わからない単語は飛ばさない**

わからない単語が出てきたらAIに聞く。「ドキュメントを読むのをやめる」のではなく「その場で解決する」。

**④ リリースノート・changelog の存在を知っておく**

多くのツールには「リリースノート」「changelog」と呼ばれる更新履歴がある。「最近何が変わったか」「自分の知識がいつ時点のものか」を確かめる一次情報として使える。

## 4-3. AI自身に「AIツールについて」聞く

「AIでAIのことを調べる」は一見おかしく聞こえるが、実際にはとても有効な方法。

### なぜ有効なのか

AIは自分自身や周辺ツールのドキュメント・仕様についての情報も学習している。難しいドキュメントを読む前に、まずAIに「わかりやすく説明して」と頼むことで、ドキュメントの理解が速くなる。

ただし注意点が1つ。

> **AIの回答には学習データのカットオフがある。** 進化の速いAIツール自身の最新機能については、AIの回答が一番古びやすい。AIの回答を「理解のための補助」として使い、最終確認は公式ドキュメントで行う。

### 具体的な使い方

ドキュメントを読んでいてわからない部分をコピーして、AIに貼り付ける。

```
以下のドキュメントの内容を、エンジニア歴3年未満の人向けにわかりやすく説明してください。

[ドキュメントの文章をここに貼り付ける]
```

または、特定の機能について概要を聞く。

```
[ツール名] の [機能名] は何をするものですか？
具体的な使用例も教えてください。
```

### ツールに「ツール自身のこと」を聞く

Gemini CLI にも Claude Code にも、ツール自身の使い方を答えさせる方法がある（ヘルプコマンドや、そのまま「あなたの使い方を教えて」と聞く）。手元で動いているツールに直接聞けるのは、CLIツールならではの調べ方。各トラックの章で実際に使う。

## 4-4. 情報の鮮度と信頼性を見極める

AIツールの情報を調べるときは、「いつ書かれたか」が重要。

### 鮮度を確認するポイント

| 確認すること | 方法 |
|---|---|
| 記事の公開・更新日 | ページ上部か下部に表示されていることが多い |
| 対象バージョンの明記 | 「◯◯ のバージョンX時点」などの記述があるか |
| URLに年号が入っているか | `/2023/` のようなURLは古い可能性がある |

### 信頼性を確認するポイント

| 確認すること | 判断基準 |
|---|---|
| 書いた人・組織 | 公式か、信頼できる個人や組織か |
| 参照元が明記されているか | 「公式ドキュメントによると」などの引用があるか |
| 実際に試した結果があるか | 「やってみたら動いた」という実証があるか |

### 情報源の信頼度を序列で持っておく

迷ったときは、上にあるものを優先する。

```
1. 公式ドキュメント・公式リリースノート
2. 開発元の公式ブログ・公式リポジトリのREADME
3. 実証つきの技術記事（日付・バージョン明記）
4. AIの回答
5. 日付不明・出典不明の記事やSNS投稿
```

AIの回答が4番目なのがポイント。**便利だが、一次情報ではない。**

### AIに情報の確認を頼む

```
以下の情報は現在でも正確ですか？また、この情報はいつ頃のものですか？

[確認したい情報を貼り付ける]
```

AIも間違えることがあるので、重要な情報は複数の情報源で確認する習慣をつける。

## 4-5. 「わからないこと」を言語化してAIに聞く技術

AIへの質問で最もよくある失敗は、「質問が曖昧すぎる」こと。プロンプトの構造や良い質問の作り方は第5章で詳しく学ぶ。ここでは「何がわからないかをまず言葉にする」ことを身につける。

### わからないことを言語化するコツ

「何がわからないかがわからない」状態のときは、まず以下をAIに聞く。

```
[学んでいる内容] について理解できているかを確認したいです。
私の理解が正しいか確認してください。

私の理解：[自分の理解を書く]
```

自分の理解を書き出すことで、どこがずれているかが明確になる。

### 「状況・試したこと・聞きたいこと」の3点セット

エラーや詰まりごとを聞くときは、次の3点を揃えると回答の精度が一気に上がる。

```
状況：[何をしようとして、何が起きたか。エラーメッセージはそのまま貼る]
試したこと：[自分が試したことと、その結果]
聞きたいこと：[何を知りたいか]
```

これは人間の先輩に質問するときと同じ作法。AIにも効く。

## やってみよう

### ① 自分のトラックの公式ドキュメントを読む

- Gemini トラック：[Gemini CLI の GitHub リポジトリ](https://github.com/google-gemini/gemini-cli) の README
- Claude トラック：[Claude Code ドキュメント](https://code.claude.com/docs) のトップページ

を開いて読む。読んだ後、以下を書き出してみよう。

- わかったこと
- わからなかったこと
- もっと知りたいこと

---

### ② ブログ記事と公式ドキュメントを比較する

「[自分が使うツール名] 使い方」で検索して出てきたブログ記事を1つ選ぶ。

その記事に書かれている内容を1つ取り上げて、公式ドキュメントに同じ情報があるか確認する。

- 情報は一致しているか？
- 記事はいつ書かれたものか？
- 食い違っていた場合、どちらを信じるべきか？（→ 4-4 の序列）

---

### ③ わからないことを1文で書いてAIに聞く

この章を読んで、まだよくわからないことを1文で書き出す。

書けたら、4-5 で学んだ「良い質問」の形式に整えてAIに送ってみよう。

---

### ④ AIの回答に「それはいつの情報か」を聞く

何かAIに質問して回答をもらったら、続けてこう聞く。

```
その情報はいつ頃のものですか？最新の状況と変わっている可能性はありますか？
```

回答を見て、情報の鮮度への意識を持つ習慣をつける。

---

### ⑤ AIの回答をわざと一次情報で検証する

AIに「[自分が使うツール] の最新バージョンの新機能を教えて」と聞いてみる。

その回答を、公式のリリースノート・changelog と突き合わせる。一致しているか？古い情報が混じっていないか？

**「AIの回答を一次情報で検証する」一連の動きを、ここで一度フルでやっておく。** この動きが身についていれば、このコンテンツを卒業した後も自走できる。
