## 1-1. LLM（大規模言語モデル）とは何か

### なぜ仕組みを知る必要があるのか

AIを使いこなすには、AIが「どうやって動いているか」のおおまかなイメージを持っておくことが重要。

全部理解する必要はない。でも「なんとなく仕組みがわかってる」状態と「全くわからない」状態では、AIとの対話の質が大きく変わる。なぜなら、仕組みを知っていると「なぜAIがこういう答えを返したのか」「どんな頼み方をすれば意図通りに動くか」が推測できるようになるから。

### LLMとは

ChatGPT、Gemini、Claude といった会話できるAIは、**LLM（Large Language Model＝大規模言語モデル）** と呼ばれる技術をベースにしている。

LLMは一言でいうと、**「大量のテキストから言葉のパターンを学んだ予測エンジン」**。

インターネット上の膨大な文章（書籍・ニュース・コード・論文など）を読み込み、「この文脈のあとにはどんな言葉が続くか」を繰り返し学習している。質問を投げると、学習したパターンをもとに「最も自然な続き」を生成して返す。

### 「次の言葉の予測」がなぜ会話になるのか

LLMの動作を一段だけ掘り下げる。

LLMは文章を **トークン**（単語や文字のかたまり）という単位に分解して処理する。そして「ここまでのトークン列に続く、次のトークンは何か」を確率で予測する。

```
入力：「日本の首都は」
予測：「東京」85%・「東」10%・「京都」3%・…
```

これを1トークンずつ繰り返すことで文章ができあがる。**1文字ずつ「続きを当てるゲーム」を超高速でやっている**、というイメージが近い。

この仕組みを知っていると、次の2つが腑に落ちる。

- **同じ質問でも毎回答えが変わる**：予測は確率なので、選ばれるトークンが毎回少しずつ違う
- **もっともらしい嘘をつく**：「正しいか」ではなく「自然な続きか」で生成しているため、自然だが間違っている文章が出てくることがある

### LLMが「知っている」ということの意味

ここで重要な点がある。LLMは「知識を持っている」というより、「知識があるように見える文章を生成できる」と理解した方が正確。

- 学習データに含まれていた情報は答えられる
- 学習データの**カットオフ日時以降**の情報は持っていない
- 確信がなくても、もっともらしい文章を生成することがある（これを**ハルシネーション**という）

> **用語｜カットオフ：** AIの学習データの締め切り日時。例えばカットオフが2024年4月なら、それ以降に起きた出来事や更新された仕様は知らない。
> **用語｜ハルシネーション：** 事実ではないことを自信を持って正しいかのように生成する現象。AIは「知らない」と言わず、もっともらしい嘘をつくことがある。

なお、最近のAIツールの多くは **Web検索機能** を組み合わせることで、カットオフ以降の情報も調べられるようになっている。ただし「検索した結果の要約」にも間違いは混じる。仕組みが変わっても「鵜呑みにしない」原則は変わらない。

この特性を理解しておかないと、AIの回答を鵜呑みにして間違った情報を使ってしまう。AIの出力は「参考情報」として扱い、重要な判断は自分で確認する癖をつけることが大切。

## 1-2. AIにできること・できないこと

### エンジニアの業務でAIが得意なこと

| できること | 具体例 |
|---|---|
| コードの生成 | 「ボタンを押したらリストに追加するJSを書いて」 |
| コードの説明 | 「このコードが何をしているか説明して」 |
| バグの原因調査 | 「このエラーメッセージの原因を教えて」 |
| リファクタリング案の提示 | 「このコードをもっと読みやすくして」 |
| ドキュメントの下書き | 「この仕様をREADMEにまとめて」 |
| 調査・情報収集の補助 | 「〇〇の使い方を教えて」 |
| 壁打ち相手 | 「この設計の問題点を指摘して」 |

### AIが苦手なこと・任せてはいけないこと

| できないこと・任せてはいけないこと | 理由 |
|---|---|
| 最新情報の提供 | 学習データに時差があるため（Web検索併用でも要確認） |
| 社内固有のシステムや仕様の把握 | 学習データに含まれていないため |
| 生成したコードの動作保証 | AIは動くと思って出力するが、実際には動かないこともある |
| セキュリティ上の最終判断 | リスクの判断は人間が行う必要がある |
| 機密情報の取り扱い | 渡した情報がどう使われるかはポリシー次第（第2章で詳しく扱う） |
| 「何を作るべきか」の決定 | ビジネス上の判断・優先順位づけは人間の仕事 |

**AIを「賢いアシスタント」として使うのは正しい。でも「全部任せられる自律した存在」として扱うのは間違い。** 最終的な判断と責任は常に人間にある。

## 1-3. このコンテンツで使うAIツール

このコンテンツでは、次のどちらかのトラックを選んで進める。**どちらを選んでも学ぶ内容のゴールは同じ。**

### Gemini トラック

| ツール | 役割 |
|---|---|
| **Gemini Enterprise**（Google Workspace 向け） | ブラウザで使うWeb UI。気軽な質問・会話に使う |
| **Gemini CLI** | ターミナルから Gemini を操作できる公式オープンソースツール。ファイルを渡したり自動化したりできる |

### Claude トラック

| ツール | 役割 |
|---|---|
| **Claude**（claude.ai） | ブラウザで使うWeb UI。気軽な質問・会話に使う |
| **Claude Code** | ターミナルで動く公式のエージェント型コーディングツール。会話だけでなく、ファイルの読み書きやコマンド実行までAI自身が行う |

> **用語｜エージェント型コーディングツール：** 「質問に答える」だけでなく、ファイルを読む・書き換える・コマンドを実行するといった操作をAI自身が行うツール。人間は操作のたびに許可を出し、結果をレビューする役割になる。

### 「チャット型」と「エージェント型」の違い

2つのトラックの道具立てには性格の違いがある。これは優劣ではなく、人間の関わり方の違い。

| | チャット型（Web UI など） | エージェント型（Claude Code など） |
|---|---|---|
| AIの出力 | 文章・コードを「提案」する | ファイル編集・コマンド実行まで「実行」する |
| 人間の役割 | 提案をコピーして自分で反映する | 実行前に許可を出し、実行後に差分をレビューする |
| 向く場面 | 質問・学習・下書き | 実際の開発作業 |

どちらの形でも、**「AIの出力を評価して採否を決めるのは人間」** という原則は変わらない。エージェント型はAIが直接ファイルを変更するぶん、レビューの重要性がさらに上がる。

> 業務でAIを使う場合は、入力したデータがどう扱われるかが最重要になる。各ツールのデータポリシーは第2章で扱う。「企業向けプランだから何を入力しても安全」ではない点も含めて、第2章で必ず確認すること。

## 1-4. なぜエンジニアがAIを使うのか

### 「書く時間」より「考える時間」を増やすため

コードを書く作業の中には、調べればわかること・決まったパターンがあること・繰り返しの作業が多く含まれている。

AIはこういった「すでに答えがある作業」を高速に処理することが得意。AIに任せることで、**「どう設計するか」「何を作るか」「どう判断するか」という人間が考えるべき部分に集中できる**。

### スピードの問題ではない

AIを使う理由を「速くなるから」と理解している人は多い。それは正しいが、半分しか正しくない。

重要なのは**「試行回数が増える」**こと。

コードの初稿をAIに出させて、それをレビューして修正する。このサイクルを繰り返すことで、1人で全部考えて書くより多くの選択肢を検討できる。「いいものを作る」ために使うのがAIの本質的な価値。

### 「AIで楽になった時間」を何に使うか

はじめにでも書いたとおり、ここが分かれ道になる。

- 楽になった時間で**理解を深める**エンジニア → AIで成長が加速する
- 楽になった時間を**何もせず流す**エンジニア → AIに置き換えられていく

このコンテンツの演習で「AIに聞いて終わり」ではなく「なぜそうなるかを確認する」手順が多いのは、この分かれ道を意識しているから。

## やってみよう

> この章の演習は **ブラウザのWeb UI** で実行する。CLI のセットアップは第3章で行うので、今はまだ不要。
>
> - Gemini トラック：[gemini.google.com](https://gemini.google.com) を Google Workspace のアカウントで開く
> - Claude トラック：[claude.ai](https://claude.ai) を自分のアカウントで開く

### ① AIに自己紹介させる

Web UI を開いて、以下を送ってみよう。

```
あなたはどういう仕組みで動いていますか？
小学生でもわかるように説明してください。
```

受け取った回答を読んで、この章で学んだ「LLMの仕組み」と照らし合わせてみる。

---

### ② 同じ質問を2回送って、答えの揺れを観察する

新しい会話を2つ開いて、まったく同じ質問を送る。

```
JavaScriptの配列のメソッドを3つ教えてください。
```

2つの回答を見比べてみよう。選ばれるメソッドや説明の仕方が微妙に違うはず。これが 1-1 で学んだ「確率による予測」の現れ。**AIの回答は毎回同じではない**ことを体感しておく。

---

### ③ 「AIにできないこと」を自分でリストアップする

自分の今の業務や開発作業を思い浮かべながら、「これはAIに任せられないな」と思うことを3つ書き出してみよう。

書き出したら、AIにこう聞いてみる。

```
エンジニアの業務の中で、AIが苦手なこと・任せてはいけないことを教えてください。
```

自分のリストと回答を比べて、気づかなかった観点がないか確認する。

---

### ④ AIの回答の「限界」を体験する

以下のように聞いてみよう。

```
あなたの学習データはいつまでの情報ですか？
```

続けて、カットオフより後の出来事（最近のニュースなど）について聞いてみる。Web検索機能が動く場合と動かない場合で、回答がどう変わるかも観察する。

「AIが持っている情報の範囲」と「検索で補っている範囲」の区別を体感する。
