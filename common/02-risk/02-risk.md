# 第2章：AIを使う前に知るべきリスク

**目安時間：4時間**

> この章はツールを設定する前に必ず読むこと。知らずに使い始めると、取り返しのつかない情報漏洩につながる可能性がある。

---

## この章で学ぶこと

- AIに入力した情報がどう扱われるかを理解する
- 個人向けプランと企業向けプランのデータポリシーの違いを把握する
- AIに渡してはいけない情報の種類を知る
- GitHubにコミットしてはいけないものを理解する
- エージェント型ツール特有のリスク（コマンド実行・権限管理）を知る
- 人間が判断すべきことと、AIに任せてよいことの線引きを持つ

---

## 2-1. AIに渡した情報はどこへ行くのか

AIとの会話は「ただのチャット」ではない。入力した内容はサーバーに送信され、処理される。

多くのAIサービスでは、送信された内容が以下のいずれかの用途に使われる可能性がある。

| 用途 | 内容 |
|---|---|
| モデルの改善・学習 | 入力データをAIの性能向上に使用する |
| サービス品質の向上 | 不適切な出力のチェックなどに人間がレビューする |
| ログの保存 | 一定期間、会話内容がサーバーに保存される |

「AIに話しかけること」は、情報をインターネット上のサービスに送信することと同じ。この前提を忘れずに使うことが重要。

---

## 2-2. 個人向けと企業向けでデータポリシーは違う

AIサービスには複数のプランがあり、**入力データが学習に使われるかどうか** がプランによって異なる。これはツールを問わず共通の構図。

### 共通の構図

| プランの種類 | 学習利用の傾向 | 管理機能 |
|---|---|---|
| 個人向け無料プラン | 学習に使われる可能性がある（設定で変えられる場合もある） | なし |
| 個人向け有料プラン | サービス・設定によって異なる | なし |
| **企業向けプラン** | **学習に使われないことが契約で保証されるのが一般的** | 管理者が利用状況を管理できる |

### このコンテンツで使うツールの場合

| トラック | 使うプラン | ポイント |
|---|---|---|
| Gemini | Gemini Enterprise（Google Workspace） | 入力データはモデル学習に使われない。管理者が利用管理できる |
| Claude | Claude Pro / Max / Team など | 個人向けプランは**学習利用に関する設定がアカウントにある**ため必ず自分で確認する。Team / Enterprise や API はビジネス利用前提のポリシーが適用される |

> **ポリシーは変わる。** この表は執筆時点の整理であり、正確な内容は必ず各社の公式ページで確認すること（このコンテンツの第4章で「公式情報の確認の仕方」自体を学ぶ）。
>
> - Google：Google Workspace の Gemini に関するプライバシー説明
> - Anthropic：Privacy Center・利用規約（Consumer / Commercial）

業務でAIを使う場合に企業向けプランを使う最大の理由はここにある。**会社の情報がAIの学習データとして外部に漏れない**という契約上の保証があるかどうか。

ただし、これは「何でも入力していい」という意味ではない。

> 企業向けプランであっても、入力データは提供元のサーバーで処理される。「学習に使われない」と「完全に安全」は別の話。機密情報の扱いは次のセクションで整理する。

## 2-3. AIに渡してはいけない情報

以下の情報は、たとえ企業向けプランであっても AIに渡さない。

### 絶対に渡してはいけないもの

| 種類 | 具体例 |
|---|---|
| 認証情報 | APIキー・パスワード・トークン・秘密鍵 |
| 個人情報 | 氏名・住所・電話番号・メールアドレス・マイナンバー |
| 機密性の高い社内情報 | 未公開の財務情報・M&A情報・顧客の個人データ |
| 規制対象データ | 医療情報・金融情報など法的に保護されているデータ |

### 渡す前に考えるべきもの

| 種類 | 判断基準 |
|---|---|
| 社内のソースコード | 公開しても問題ないか・競合に見られて困る内容を含まないか |
| 顧客に関する情報 | 個人が特定できる情報が含まれていないか |
| 社内の仕様書・設計書 | 機密指定されていないか |

迷ったときの判断基準：**「このテキストをインターネットに公開しても問題ないか？」** と考える。問題があるなら渡さない。

### 会社のルールが最優先

このコンテンツの判断基準は一般論。**所属企業・客先にAI利用のガイドラインがある場合は、必ずそちらに従う。** 「ガイドラインがあるか知らない」状態で使い始めるのが一番危ない。使う前に確認する。

## 2-4. コードをAIに渡す際のリスク

コードを渡してレビューや修正を頼むことは、AIの得意な使い方のひとつ。ただし、コードの中に機密情報が混入していることがある。

### よくある危険なパターン

```javascript
// 絶対にやってはいけない例
const API_KEY = "AIzaSyXXXXXXXXXXXXXXXXXXXXXXXX";
const DB_PASSWORD = "mypassword123";

fetch(`https://api.example.com/data?key=${API_KEY}`);
```

コード内に直接書かれた認証情報はそのままAIに送信される。

### 正しい扱い方

認証情報は `.env` ファイルに分離して管理し、コードには書かない。

> **用語｜`.env` ファイル：** APIキーやパスワードなど機密情報を記述する専用ファイル。コードとは別ファイルに分離することで、誤ってGitHubに公開するリスクを減らせる。

```javascript
// 正しい例：環境変数から読み込む
const API_KEY = process.env.API_KEY;
```

```bash
# .env ファイル（AIには渡さない・GitHubにもコミットしない）
API_KEY=AIzaSyXXXXXXXXXXXXXXXXXXXXXXXX
DB_PASSWORD=mypassword123
```

AIにコードを渡す前に、認証情報や個人情報が含まれていないかを必ず確認する。

### エージェント型ツールでは「渡さないつもりでも読まれる」ことがある

Claude Code のようなエージェント型ツールは、作業に必要だと判断したファイルを**自分で読みに行く**。プロジェクトフォルダの中に `.env` や顧客データのファイルがあると、こちらが貼り付けなくても読まれる可能性がある。

対策は2つ。

1. **プロジェクトフォルダに機密ファイルを置かない**（置く場合はツール側の除外設定を使う。設定方法は各トラックの第3章・第5章で扱う）
2. **ツールがファイルを読む・コマンドを実行する前の確認（許可プロンプト）を雑に承認しない**（2-7で詳しく扱う）

## 2-5. GitHubにコミットしてはいけないもの

GitHubのパブリックリポジトリは世界中に公開される。プライベートリポジトリでも、誤って公開設定にしてしまうリスクや、アカウントが侵害されるリスクがある。

### コミットしてはいけないファイル・情報

| ファイル・情報 | 理由 |
|---|---|
| `.env` ファイル | APIキーやパスワードが含まれる |
| `*_secret.json` などの認証情報ファイル | 同上 |
| `node_modules/` | 容量が大きく不要・`package.json` から復元できる |
| ビルド成果物（`dist/`, `build/`） | ソースから生成できる |
| OS・エディタが生成するファイル | `.DS_Store`, `.vscode/settings.json` など |

### `.gitignore` で守る

`.gitignore` ファイルに記載されたファイルは Git の管理対象から除外される。プロジェクトを作ったら最初に設定する。

```bash
# .gitignore の基本的な記述例
.env
.env.local
node_modules/
dist/
.DS_Store
```

### 漏らしてしまったときの対応

> 一度コミットしてGitHubにプッシュしてしまった認証情報は、**その瞬間に漏洩したとみなす**。履歴から削除しても、すでにスキャンされている可能性がある。

公開リポジトリは、認証情報を自動収集するボットに常時スキャンされている。プッシュから悪用までは分単位で起こりうる。

漏らしたときにやることは1つ。**「履歴の削除」ではなく「キーの無効化・再発行」**。

1. 漏れたAPIキー・トークンを発行元のサービスで即座に無効化する
2. 新しいキーを発行して `.env` に入れ直す
3. その後で履歴の整理を検討する（順番を逆にしない）

### AIにコミットを任せるときも同じ

エージェント型ツールは `git add` や `git commit` まで実行できる。**AIがコミットする場合でも、コミット前チェックの責任は人間にある。** `git status` と `git diff` で「何がコミットされようとしているか」を自分の目で確認する習慣を、第3章以降で徹底的に身につける。

## 2-6. AIが生成したコードの著作権・ライセンス問題

AIが生成したコードは学習データのパターンから生成される。そのため、学習データに含まれたオープンソースコードに似たコードが出力される場合がある。

現時点では、AIが生成したコードの著作権については法的にまだ議論が続いている部分もある。ただし、以下の点は意識しておくべき。

- 商用プロジェクトで使う場合、生成されたコードが特定のライセンス（GPL など）に抵触しないか確認することが望ましい

> **用語｜GPL（GNU General Public License）：** オープンソースライセンスの一つ。GPLのコードを使ったソフトウェアはソースコードの公開義務が生じる場合がある。商用利用には注意が必要。

- 生成されたコードをそのままコピー&ペーストするだけでなく、**内容を理解した上で使う**ことが重要
- 「AIが生成したから責任はない」は通用しない。コードの品質と法的問題の責任は使った人間にある

## 2-7. エージェント型ツール特有のリスク

Claude Code や Gemini CLI のエージェント機能は、ファイルの編集やコマンドの実行までAIが行う。便利なぶん、チャット型にはないリスクがある。**Claudeトラックに進む人は特に必読。Geminiトラックでも考え方は同じなので読んでおく。**

### リスク① 意図しない操作の実行

AIが「よかれと思って」ファイルを消したり、想定外のコマンドを実行したりする可能性がある。

エージェント型ツールには操作前に人間の許可を求める仕組み（許可プロンプト）がある。これが最後の防衛線。

**やってはいけないのは「全部許可」で思考停止すること。** 許可を求められたら、最低限これだけは確認する。

- どのファイルを変更しようとしているか
- どんなコマンドを実行しようとしているか（特に `rm`・`git push`・インストール系）
- それは今の依頼に必要な操作か

### リスク② 権限の広げすぎ

「毎回確認されるのが面倒だから」と確認なしの自動実行を広く許可していくと、事故のときの被害が大きくなる。

> **用語｜最小権限の原則：** 与える権限は「必要な分だけ」にするという考え方。AIへの許可も、読み取り系は許可してよいが、削除・送信系は毎回確認する、のように段階をつける。

| 操作の種類 | 自動許可の目安 |
|---|---|
| ファイルの読み取り・検索 | 許可してよいことが多い |
| ファイルの編集 | プロジェクト内なら許可しても、diff の確認は必ず行う |
| `git commit` などの記録系 | 慣れるまでは毎回確認 |
| 削除・`git push`・外部送信系 | **常に毎回確認** |

### リスク③ プロンプトインジェクション

AIが読み込むファイルやWebページの中に、「これまでの指示を無視して◯◯せよ」のような**AIに対する攻撃的な指示**が仕込まれていることがある。AIがそれを本物の指示と混同して実行してしまう攻撃を **プロンプトインジェクション** と呼ぶ。

> **用語｜プロンプトインジェクション：** AIに渡されるデータの中に悪意ある指示を埋め込み、AIの動作を乗っ取ろうとする攻撃。

エンジニア3年未満の段階で対策の詳細まで覚える必要はない。ただ、次の2点だけ頭に入れておく。

- **出どころの怪しいファイル・リポジトリ・Webページを安易にAIに読ませない**
- AIが急に依頼と関係ない操作（外部への送信など）をしようとしたら、許可せず内容を確認する

### このコンテンツでの方針

各トラックの第3章・第5章で、許可設定を実際に自分で管理する。「便利さ」と「安全」のバランスを自分で決められるようになることが目標。

## 2-8. 人間が管理すべき判断とAIに任せてよい判断の線引き

AIは強力なアシスタントだが、すべての判断を任せることはできない。

### AIに任せてよいこと

- コードの初稿を生成する
- 書いたコードの問題点を指摘させる
- ドキュメントの下書きを作る
- エラーメッセージの原因を調べる
- 複数の実装方法の選択肢を出させる

### 人間が必ず判断すること

| 判断 | 理由 |
|---|---|
| セキュリティに関わる設計 | AIは脆弱性を見落とすことがある |
| 本番環境へのデプロイ | 影響範囲が大きく取り返しがつかない |
| 顧客・ユーザーへの影響がある変更 | AIは文脈やビジネス上の影響を完全には理解できない |
| 生成されたコードが正しいかどうかの最終確認 | AIは間違いを自信満々に返すことがある |
| 何をAIに渡すかの判断 | この章で学んだこと |
| AIにどこまでの操作を許可するかの判断 | 2-7 で学んだこと |

**AIは「作業を速くするツール」であり、「責任を引き受けるツール」ではない。** 最終的な判断と責任は常に人間にある。

## やってみよう

> この章の演習は **ブラウザのWeb UI** で実行する。CLI のセットアップは第3章で行うので、今はまだ不要。

### ① 使うツールのデータポリシーを自分で確認する

自分のトラックのデータポリシーページを自分で開いて読んでみよう。

- Gemini トラック：Google Workspace の Gemini に関するプライバシーページ
- Claude トラック：Anthropic の Privacy Center と、自分のアカウントのプライバシー設定画面

読んだ後、AIに以下のプロンプトを送って、答えを照らし合わせてみる。

```
[自分が使うサービス名] のデータポリシーについて教えてください。
入力したデータが学習に使われるかどうかも含めて説明してください。
```

AIの回答と公式ページの記載が食い違っていたら、**公式ページの方を信じる**。この「AIの回答を一次情報で検証する」動きは第4章で本格的に身につける。

---

### ② 「渡してはいけない情報」を自分の業務で考える

自分が今関わっているプロジェクトや業務を思い浮かべながら、以下を書き出してみよう。

- AIに渡しても問題ない情報
- AIに渡してはいけない情報（その理由も書く）
- 所属企業・客先のAI利用ガイドラインの有無（知らなければ確認する）

---

### ③ ダミーの `.env` ファイルを作って中身を確認する

実際の認証情報ではなく、ダミーの値で `.env` ファイルを作ってみよう。

```bash
# .env（ダミー）
API_KEY=dummy_api_key_12345
DB_PASSWORD=dummy_password
SECRET_TOKEN=dummy_secret_token
```

このファイルをGitHubにコミットするとどうなるかを想像する。次の章でセットアップを行う際に、これを `.gitignore` で除外する手順を実際に確認する。

---

### ④ 「許可を出すかどうか」のシミュレーション

エージェント型ツールが以下の操作の許可を求めてきたとする。それぞれ「許可する／内容を確認してから判断／許可しない」のどれにするか、理由つきで考えてみよう。

1. `index.html` を読み取る
2. `style.css` を編集する
3. `rm -rf node_modules` を実行する
4. `.env` を読み取る
5. `git push origin main` を実行する

（考え方の例：1は読み取りのみで低リスク。4は機密ファイルなので原則拒否。3と5は内容と状況の確認が必要）
