# 学習コンテンツ 全体章立て

AIを使ってTo-Doリストアプリをバイブコーディングで作るまでのロードマップ。

**総学習時間：約46時間（1日8時間 × 約6日間）**

---

## 2トラック構成

このコンテンツは **共通章** と **ツール別トラック** で構成する。

```
common/   … ツールに依存しない章（はじめに・第1章・第2章・第4章）
gemini/   … Gemini Enterprise ＋ Gemini CLI で学ぶトラック（第3・5・6・7・8章）
claude/   … Claude ＋ Claude Code で学ぶトラック（第3・5・6・7・8章）
```

- 学習者はどちらか一方のトラックを選んで進める（ゴールは同一：To-Doアプリの完成）
- 読む順番：00 → 01 → 02 →（トラック）03 → 04 →（トラック）05 → 06 → 07 → 08
- 共通章にはツール固有の手順を書かない。ツール固有の内容は必ずトラック側に置く

## 各章共通の構成

各章は以下の4パートで構成する。

1. **解説**：概念・仕組み・背景を説明する
2. **やってみよう**：実際に手を動かして確認する（読むだけで終わらせない）
3. **理解度チェック**：自分の言葉で答えられるかを確認する設問（セクション参照つき）
4. **もっと深く知るために**：公式ドキュメントへのリンク・AIに聞けるサンプルプロンプト・次に調べるキーワード

## はじめに：このコンテンツを読む前に｜common

**ゴール：** AIがエンジニアのキャリアに与える影響を正しく理解した上で、学習を始める

- AIは脳死でコードを承認するエンジニアを淘汰するツールである
- 同時に、理解に使う時間を増やせるエンジニアにとっては最強の武器である
- このコンテンツで身につけるのは「使い方」ではなく「判断力」
- 2トラックの構成と選び方・用語集

## 第1章：AIとは何か｜common｜3時間

**ゴール：** AIに対する「よくわからない」を取り除く

### 解説
- 1-1. LLM（大規模言語モデル）とは何か（トークン・確率予測のイメージまで）
- 1-2. AIにできること・できないこと
- 1-3. このコンテンツで使うAIツール（チャット型とエージェント型の違い）
- 1-4. なぜエンジニアがAIを使うのか

### やってみよう
- AIに「あなたはどういう仕組みで動いていますか？」と聞いてみる
- 同じ質問を2回送って答えの揺れを観察する
- 「AIにできないこと」を自分なりにリストアップして、AIに確認してもらう
- カットオフとWeb検索の範囲を体感する

## 第2章：AIを使う前に知るべきリスク｜common｜4時間

**ゴール：** リスクを理解した上で使える判断軸を持つ。ツールを設定する前に必ず読む章

### 解説
- 2-1. AIに渡した情報はどこへ行くのか（データの扱いと学習利用の仕組み）
- 2-2. 個人向けプランと企業向けプランのデータポリシーの違い（Gemini / Claude 両方）
- 2-3. AIに渡してはいけない情報（個人情報・機密情報・社内データ・認証情報）
- 2-4. コードをAIに渡す際のリスク（APIキー・`.env`・エージェント型は「読まれる」リスクも）
- 2-5. GitHubにコミットしてはいけないもの・漏らしたときの対応
- 2-6. AIが生成したコードの著作権・ライセンス問題
- 2-7. エージェント型ツール特有のリスク（許可プロンプト・最小権限・プロンプトインジェクション）
- 2-8. 人間が管理すべき判断とAIに任せてよい判断の線引き

### やってみよう
- 使うツールの公式データポリシーページを自分で開いて読む
- 「自分の業務でAIに渡してはいけない情報」を書き出してみる
- ダミーの `.env` ファイルを作り、何を書いてはいけないかを確認する
- 「許可を出すかどうか」のシミュレーション

## 第3章：開発環境のセットアップ｜トラック別｜6時間

**ゴール：** VS Code・ターミナル・Git / GitHub・AI CLIツールをすべて使える状態にする

### 解説（両トラック共通の流れ）
- 3-1. Visual Studio Code のインストールと基本設定
- 3-2. ターミナルの基本操作（ファイル操作・ディレクトリ移動）
- 3-3. Git のインストールと初期設定
- 3-4. GitHub アカウントの作成とリポジトリの作成
- 3-5. 基本的な Git 操作（init・add・commit・push・diff）
- 3-6. `.gitignore` の書き方（第2章で学んだリスクを実際に対策する）

### トラック別
| | Gemini トラック | Claude トラック |
|---|---|---|
| 3-7 | Gemini CLI のインストールと認証 | Claude Code のインストールと認証 |
| 3-8 | Gemini CLI の起動・ワンショット実行 | Claude Code の起動・スラッシュコマンド・許可プロンプト体験 |
| 3-9 | Web UI と CLI の使い分け | Web UI（claude.ai）と Claude Code の使い分け |

### やってみよう
- VS Code でフォルダを開き、ターミナルを起動する
- `git init` から `git push` まで一通り実行する
- `.gitignore` に `.env` を追加してコミット対象から外れることを確認する
- AIツールで最初のメッセージを送る／（Claude）ファイル作成と許可・diff確認を体験する
- エラーをAIに解決させる体験をする

## 第4章：情報を自分で集める力を身につける｜common｜4時間

**ゴール：** 公式ドキュメントの読み方とAIを使った情報収集の手段を獲得する

### 解説
- 4-1. 公式ドキュメントはなぜ読むべきか（ブログ・記事との違い）
- 4-2. 自分が使うツールの公式ドキュメントの構造と読み方（リリースノートの存在も）
- 4-3. AI自身に「AIツールについて」聞く（逆説的だが有効な方法）
- 4-4. 情報の鮮度と信頼性を見極める（情報源の信頼度の序列）
- 4-5. 「わからないこと」を言語化してAIに聞く技術（状況・試したこと・聞きたいことの3点セット）

### やってみよう
- 自分のトラックの公式ドキュメントを開き、目次から把握する
- ブログ記事と公式ドキュメントで同じトピックを調べ、内容の違いを比較する
- 「自分が今わからないこと」を1文で書いて、AIに聞いてみる
- AIの回答を一次情報（リリースノート）で検証するフルサイクルを一度回す

## 第5章：AIをハンドリングする｜トラック別｜8時間

**ゴール：** プロンプトを投げるだけでなく、AIを意図通りに動かせるようになる

### 解説（考え方は両トラック共通）
- 5-1. プロンプトの構造（役割・指示・制約・文脈／エージェント型は＋作業範囲）
- 5-2. 常時有効な指示の設定（Gemini：Gems・`GEMINI.md`／Claude：`CLAUDE.md`・`/init`）
- 5-3. コンテキストの渡し方（ファイル・仕様・前提条件）
- 5-4. トラック別のコンテキスト操作
  - Gemini：`--context`・パイプ・複数ファイル
  - Claude：`@`ファイル参照・`/clear`・`/compact`・会話とCLAUDE.mdの分担
- 5-5.（Claude）許可設定の管理（`/permissions`・settings・最小権限・計画→承認→実行）
- 5-6. AIの回答を引き出すコツ（質問の仕方・反復の仕方）
- 5-7. AIが間違えやすいパターンを知る（エージェント型は「やりすぎ」も）
- 5-8. 詰まったときのチェックリスト

### やってみよう
- 同じ質問を「役割なし」と「役割あり」で送り、回答の違いを比べる
- 常時指示ファイルを設定して、回答スタイルを変えてみる
- 仕様ファイルを渡し、それをもとに回答させる
- 意図的に曖昧なプロンプトを送って、どんな誤解が起きるかを確認する
- 間違った回答を修正指示で直させる一往復を体験する
-（Claude）許可設定の棚卸し・計画→承認→実行の通し体験

---

## 第6章：バイブコーディング入門｜トラック別｜7時間

**ゴール：** AIとの対話でコードを作るサイクルを身につけ、出力を評価できるようになる

### 解説
- 6-1. バイブコーディングとは何か（本当に必要なのは「読む力」）
- 6-2. コード生成のサイクル
  - Gemini：依頼 → 出力 → 動かす → 修正（4ステップ）
  - Claude：依頼 → AI実装（許可）→ diff確認 → 動かす → 修正（5ステップ）
- 6-3. 生成されたコードを読む視点（動くか・意図通りか・理解できるか・問題ないか）
- 6-4. よくある生成ミスのパターンと見分け方（Claudeは過剰実装パターンも）
- 6-5. AIへのフィードバックの仕方（修正指示・範囲指定・中断と差し戻し）

### やってみよう
- 「ボタンを押したらアラートを出すHTML」を作らせて動かす
- 出力されたコードを1行ずつ読んで、理解できない部分をAIに聞く
- 意図的に不完全な仕様を渡してコードを生成させ、どこが欠けているか見つける
- バグを修正指示で直させる一往復を体験する
- AIのレビューと自分のレビューを比較する

## 第7章：To-Doリストアプリを作る（実践）｜トラック別｜8時間

**ゴール：** 第1〜6章で学んだことを使って、実際にアプリを完成させGitHubで管理する

### 解説
- 7-1. 作るものを決める（要件定義をAIと一緒にやる → `spec.md`）
- 7-2. GitHubリポジトリを作り `.gitignore` を設定する
- 7-3. プロジェクトの常時指示ファイルを書く（`GEMINI.md`／`CLAUDE.md`）
- 7-4. HTMLの骨格を作る
- 7-5. CSSでスタイルを整える
- 7-6. JavaScriptで機能を実装する（追加・削除・完了を1つずつ）
- 7-7. 生成されたコードを通しでレビューする（XSS・第2章のリスク観点）
- 7-8. コミット前チェック（AIにコミットを任せる練習も）
- 7-9. LocalStorage でデータを保存する
- 7-10. 改善ループ（バグ修正・機能追加をAIと一緒に行う）

### やってみよう
- AIと会話しながら要件定義書を書き、`spec.md` として保存する
- HTML→CSS→JSの順で1ステップずつ実装させる（毎回 確認→コミット）
- 動かない箇所・意図と違う箇所を自分でリストアップして修正指示を出す
- 完成したコードをコミット前レビューしてからプッシュする
- LocalStorage を追加してリロード後もタスクが残ることを確認する

---

## 第8章：アプリを育てる（デプロイ・テスト・CI）｜トラック別｜6時間

**ゴール：** 「完成後の開発」を体験し、安心して変更し続けられる仕組みを自分で作れるようになる

### 解説
- 8-1. 「完成後」こそ開発の本番（変更し続けるための仕組み）
- 8-2. GitHub Pages でデプロイする
- 8-3. テストとは何か・AIにテストを書かせる
- 8-4. テストしやすいコードへのリファクタリング（ロジックとDOMの分離・`node --test`）
- 8-5. GitHub Actions で CI を組む（わざと落とす体験も）
- 8-6. ここから先の改善ループ

### やってみよう
- GitHub Pages でアプリを公開する
- ロジックを分離するリファクタリングを実施する
- テストを書かせて実行し、テストコード自体もレビューする
- CIを組んで、わざと壊して検出させる
- 機能をひとつ追加して、仕様→実装→テスト→CI→公開のフルサイクルを自力で回す
