# 学習コンテンツ 全体章立て

Gemini Enterprise を使ってTo-Doリストアプリをバイブコーディングで作るまでのロードマップ。

**総学習時間：約40時間（1日8時間 × 5日間）**

---

## 各章共通の構成

各章は以下の3パートで構成する。

1. **解説**：概念・仕組み・背景を説明する
2. **やってみよう**：実際に手を動かして確認する（読むだけで終わらせない）
3. **もっと深く知るために**：公式ドキュメントへのリンク・Gemini に聞けるサンプルプロンプト・次に調べるキーワード

---

## はじめに：このコンテンツを読む前に

**ゴール：** AIがエンジニアのキャリアに与える影響を正しく理解した上で、学習を始める

- AIは脳死でコードを承認するエンジニアを淘汰するツールである
- 同時に、理解に使う時間を増やせるエンジニアにとっては最強の武器である
- このコンテンツで身につけるのは「使い方」ではなく「判断力」

---

## 第1章：AIとは何か　｜　3時間

**ゴール：** AIに対する「よくわからない」を取り除く

### 解説
- 1-1. LLM（大規模言語モデル）とは何か
- 1-2. AIにできること・できないこと
- 1-3. Gemini Enterprise とは何か・他のAIとの違い
- 1-4. なぜエンジニアがAIを使うのか

### やってみよう
- Gemini に「あなたはどういう仕組みで動いていますか？」と聞いてみる
- 「AIにできないこと」を自分なりにリストアップして、Gemini に確認してもらう

---

## 第2章：AIを使う前に知るべきリスク　｜　4時間

**ゴール：** リスクを理解した上で使える判断軸を持つ。ツールを設定する前に必ず読む章

### 解説
- 2-1. AIに渡した情報はどこへ行くのか（データの扱いと学習利用の仕組み）
- 2-2. Gemini Enterprise と一般向け Gemini のデータポリシーの違い
- 2-3. AIに渡してはいけない情報（個人情報・機密情報・社内データ・認証情報）
- 2-4. コードをAIに渡す際のリスク（APIキー・`.env` ファイル・社内固有のロジック）
- 2-5. GitHubにコミットしてはいけないもの
- 2-6. AIが生成したコードの著作権・ライセンス問題
- 2-7. 人間が管理すべき判断とAIに任せてよい判断の線引き

### やってみよう
- Gemini Enterprise の公式データポリシーページを自分で開いて読む
- 「自分の業務でAIに渡してはいけない情報」を書き出してみる
- ダミーの `.env` ファイルを作り、何を書いてはいけないかを確認する

---

## 第3章：開発環境のセットアップ　｜　6時間

**ゴール：** VS Code・ターミナル・Git / GitHub・Gemini CLI をすべて使える状態にする

### 解説
- 3-1. Visual Studio Code のインストールと基本設定
- 3-2. ターミナルの基本操作（ファイル操作・ディレクトリ移動）
- 3-3. Git のインストールと初期設定（`user.name` / `user.email`）
- 3-4. GitHub アカウントの作成とリポジトリの作成
- 3-5. 基本的な Git 操作（init・add・commit・push）
- 3-6. `.gitignore` の書き方（第2章で学んだリスクを実際に対策する）
- 3-7. Gemini CLI のインストールと認証設定
- 3-8. Gemini CLI の基本的な使い方（チャット・ファイル渡し・オプション）
- 3-9. Web UI（Google Workspace）と CLI の使い分け

### やってみよう
- VS Code でフォルダを開き、ターミナルを起動する
- `git init` から `git push` まで一通り実行する
- `.gitignore` に `.env` を追加してコミット対象から外れることを確認する
- Gemini CLI を起動して最初のメッセージを送る

---

## 第4章：情報を自分で集める力を身につける　｜　4時間

**ゴール：** 公式ドキュメントの読み方とAIを使った情報収集の手段を獲得する

### 解説
- 4-1. 公式ドキュメントはなぜ読むべきか（ブログ・記事との違い）
- 4-2. Gemini の公式ドキュメントの構造と読み方
- 4-3. Gemini 自身に「Gemini について」聞く（AIで情報収集する逆説的な方法）
- 4-4. 情報の鮮度と信頼性を見極める（いつの情報か・誰が書いたか）
- 4-5. 「わからないこと」を言語化してAIに聞く技術

### やってみよう
- Gemini の公式ドキュメントを開き、目次を読んで「どこに何が書いてあるか」を把握する
- ブログ記事と公式ドキュメントで同じトピックを調べ、内容の違いを比較する
- 「自分が今わからないこと」を1文で書いて、Gemini に聞いてみる
- Gemini の回答に「それはいつ時点の情報ですか？」と確認する

---

## 第5章：AIをハンドリングする　｜　8時間

**ゴール：** プロンプトを投げるだけでなく、AIを意図通りに動かせるようになる

### 解説
- 5-1. プロンプトの構造（役割・指示・制約・文脈）
- 5-2. System Instructions でガードレールを設定する
- 5-3. コンテキストの渡し方（ファイル・仕様・前提条件）
- 5-4. CLI でコンテキストファイルを渡す（`--context` / パイプ活用）
- 5-5. AIの回答を引き出すコツ（質問の仕方・反復の仕方）
- 5-6. AIが間違えやすいパターンを知る

### やってみよう
- 同じ質問を「役割なし」と「役割あり」で送り、回答の違いを比べる
- System Instructions を設定して、Gemini の回答スタイルを変えてみる
- 仕様を書いたテキストファイルを CLI で渡し、それをもとに回答させる
- 意図的に曖昧なプロンプトを送って、どんな誤解が起きるかを確認する
- 間違った回答を修正指示で直させる一往復を体験する

---

## 第6章：バイブコーディング入門　｜　7時間

**ゴール：** AIとの対話でコードを作るサイクルを身につけ、出力を評価できるようになる

### 解説
- 6-1. バイブコーディングとは何か
- 6-2. コード生成のサイクル（依頼 → 出力 → レビュー → 修正）
- 6-3. 生成されたコードを読む視点（動くかどうか・意図通りかどうか）
- 6-4. よくある生成ミスのパターンと見分け方
- 6-5. AIへのフィードバックの仕方（修正指示の出し方）

### やってみよう
- 「ボタンを押したらアラートを出すHTMLを作って」とだけ送り、コードを動かしてみる
- 出力されたコードを1行ずつ読んで、理解できない部分をGeminiに聞く
- 意図的に不完全な仕様を渡してコードを生成させ、どこが欠けているか見つける
- 生成されたコードにバグを発見したら、修正指示を出して直させる一往復を体験する

---

## 第7章：To-Doリストアプリを作る（実践）　｜　8時間

**ゴール：** 第1〜6章で学んだことを使って、実際にアプリを完成させGitHubで管理する

### 解説
- 7-1. 作るものを決める（要件定義をGeminiと一緒にやる）
- 7-2. GitHubリポジトリを作り `.gitignore` を設定する
- 7-3. プロジェクトのコンテキストを CLI で渡す（仕様ファイルをそのまま食わせる）
- 7-4. HTMLの骨格を作る
- 7-5. CSSでスタイルを整える
- 7-6. JavaScriptで機能を実装する（追加・削除・完了）
- 7-7. 生成されたコードを通しでレビューする（第2章のリスク観点も含めて）
- 7-8. コミット前チェック（コミットしてはいけないものが混じっていないか）
- 7-9. LocalStorage でデータを保存する（リロード後もタスクが残るようにする）
- 7-10. 改善ループ（バグ修正・機能追加をAIと一緒に行う）

### やってみよう
- Gemini と会話しながら要件定義書を書き、`spec.md` として保存する
- `spec.md` を CLI で渡してHTMLの初稿を生成させる
- 動かない箇所・意図と違う箇所を自分でリストアップして修正指示を出す
- 完成したコードをコミット前にレビューし、問題がないことを確認してからプッシュする
- LocalStorage を追加してリロード後もタスクが残ることを確認する
