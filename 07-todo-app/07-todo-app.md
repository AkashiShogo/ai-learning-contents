# 第7章：To-Doリストアプリを作る（実践）

**目安時間：8時間**

> **前提：** 第3〜6章を終えていること（セットアップ・AIハンドリング・バイブコーディングのサイクルを体験済み）

> これまでの章で学んだことをすべて使う。AIに指示を出し、コードを確認し、GitHubで管理しながら、動くアプリを完成させる。ただし、アプリの完成はゴールではない。ここで身につけたサイクルを、この先も自分で回し続けられるようになることが本当のゴールだ。

---

## この章で学ぶこと

- 要件定義からコード完成までの一連の流れを体験する
- `GEMINI.md` とコンテキストファイルを使ったプロジェクト管理を実践する
- 生成されたコードをレビューして問題を見つける
- コミット前チェックを習慣として身につける
- バグ修正・機能追加の改善ループを回す

---

## 7-1. 作るものを決める（要件定義をGeminiと一緒にやる）

### なぜ要件定義から始めるのか

「とりあえず作り始めて、後から直す」は、最初の方向性が間違っているとやり直しが大きくなる。AIに依頼する前に「何を作るか」を整理することで、ブレのない開発ができる。

### Gemini と一緒に要件定義をする

以下のプロンプトから始める。

```
To-Doリストアプリの要件定義を手伝ってください。

前提：
- HTML / CSS / Vanilla JS のみで実装
- 外部ライブラリは使わない
- 初心者向けの学習目的なので、シンプルな構成にする

まず、最低限必要な機能のリストを提案してください。
```

Gemini が提案してきた機能を見て、「これは必要」「これはいらない」を自分で判断する。最終的な機能リストを自分で決める。

### `spec.md` に書き出す

プロジェクトフォルダに `spec.md` を作り、決めた仕様を書く。

```markdown
# To-Doリストアプリ 仕様書

## 機能一覧
- [ ] テキスト入力欄にタスクを入力できる
- [ ] 「追加」ボタンを押すとリストにタスクが追加される
- [ ] 入力が空の場合は追加されない
- [ ] 各タスクに「完了」ボタンがあり、押すと取り消し線がつく
- [ ] 各タスクに「削除」ボタンがあり、押すとリストから消える

## 制約
- HTML / CSS / Vanilla JS のみ
- 外部ライブラリ・フレームワーク不使用
- ファイルは index.html・style.css・script.js の3ファイル構成

## 対象ブラウザ
- Chrome 最新版
```

```bash

### 最初のコミットをする

git commit -m "プロジェクトの初期設定"

```

## 7-3. プロジェクトのコンテキストを CLI で渡す

### `GEMINI.md` を書く

このプロジェクトで Gemini CLI を使うときに常に読み込まれる設定ファイル。

```markdown
# プロジェクト概要
To-Doリストアプリの学習プロジェクト。

## 技術スタック
- HTML5
- CSS3
- Vanilla JavaScript（ES6以降）

## ファイル構成
- index.html：HTMLの構造
- style.css：スタイル
- script.js：機能の実装

## コーディングルール
- 日本語でコメントを書く
- コードブロックには必ず言語タグをつける（```html, ```css, ```js）
- `var` は使わない。`const` と `let` を使う
- 外部ライブラリは使わない

## 回答形式
- 変更を加えるときは変更前・変更後を明示する
- 複数の選択肢がある場合は選択肢を提示してから進める
```

### `spec.md` を渡してHTML骨格を作らせる

```bash
gemini --context ./spec.md "この仕様書に基づいて、HTMLの骨格を作ってください。CSSとJSはまだ不要です。"
```

## 7-4. HTMLの骨格を作る

### 出力されたHTMLを確認する

Gemini が生成したHTMLを `index.html` に貼り付けて、ブラウザで開く。

確認するポイント：
- 入力欄があるか
- 追加ボタンがあるか
- リストが表示される場所があるか
- `id` や `class` が付いているか（JSで参照するために必要）

問題があれば修正指示を出す。

```
生成されたHTMLを確認しました。
以下の点を修正してください：
- [具体的な問題点]
```

### コミットする

HTMLが想定通りの構造になったらコミット。

```bash
git add index.html
git commit -m "HTMLの骨格を追加"
```

## 7-5. CSSでスタイルを整える

### CSSを依頼する

```bash
gemini --context ./index.html "このHTMLに合わせてCSSを作ってください。
シンプルで見やすいデザインにしてください。
ボタンはクリックしやすいサイズにしてください。"

### 確認ポイント

デザインの好みは人によって違うので、気に入らない部分は遠慮なく修正を依頼する。

```
ボタンの色を青にしてください。
文字サイズをもう少し大きくしてください。
```

### コミットする

```bash
git add style.css
git commit -m "CSSのスタイルを追加"
```

## 7-6. JavaScriptで機能を実装する

### 機能を1つずつ実装する

一度に全機能を依頼せず、1機能ずつ実装して確認しながら進める。

**ステップ1：タスクの追加機能**

```bash
gemini --context ./index.html --context ./spec.md \
"spec.mdの仕様に基づいて、タスクの追加機能を実装してください。
実装する機能：入力欄のテキストをリストに追加する・追加後に入力欄を空にする・入力が空の場合は追加しない"
```

動作確認してからコミット。

```bash
git add script.js
git commit -m "タスク追加機能を実装"
```

**ステップ2：完了機能**

```bash
gemini --context ./script.js "既存のコードに、タスクの完了機能を追加してください。
完了ボタンを押したら取り消し線がつく機能です。
既存のコードは変更しないで、完了機能のみ追加してください。"
```

確認してコミット。

**ステップ3：削除機能**

同じように依頼して実装・確認・コミット。

## 7-7. 生成されたコードを通しでレビューする

### コード全体を確認する

すべての機能が実装できたら、コード全体を通してレビューする。

```bash
cat ./script.js | gemini "このJavaScriptコードをレビューしてください。
以下の観点で確認してください：
1. セキュリティ上の問題（XSSなど）
2. バグや意図しない動作の可能性
3. 不要なコードや重複した処理"
```

### リスク観点でのチェック（第2章の復習）

- 機密情報が含まれていないか
- ユーザーの入力をそのままHTMLに書き出していないか（XSS）

#### XSSの例

```javascript
// 危険な書き方（ユーザー入力をそのまま innerHTML に入れる）
listItem.innerHTML = input.value;

// 安全な書き方（textContent を使う）
listItem.textContent = input.value;
```

生成されたコードで `textContent` と `innerHTML` のどちらが使われているかを確認したら、必ずGeminiに理由を聞く。

```
生成されたコードで textContent を使っていますが、innerHTML ではなく textContent を使う理由を教えてください。
セキュリティの観点からも説明してください。
```

「動いたから OK」ではなく、**なぜその書き方が選ばれたか**を理解することが、第2章で学んだリスク管理の実践になる。

レビュー結果で問題が出たら、修正指示を出して直す。

## 7-8. コミット前チェック

コードをプッシュする前に必ず確認する。

### チェックリスト

```
□ .env などの機密ファイルが .gitignore に含まれているか
□ コード内にAPIキー・パスワードなどが直書きされていないか
□ git status で意図しないファイルがステージングされていないか
□ ブラウザで全機能が動くことを確認したか
```

### コミットメッセージのルール

後から見てわかるメッセージを書く。

```bash
# 悪い例
git commit -m "修正"
git commit -m "update"

# 良い例
git commit -m "タスク削除ボタンを押したときのバグを修正"
git commit -m "CSSのボタンデザインを変更"
```

### Gemini にコミットメッセージを書かせる（おまけ）

せっかく Gemini CLI があるので、変更内容を渡してコミットメッセージを生成させてみよう。

```bash
git diff --staged | gemini -p "このdiffをもとに、日本語で簡潔なgitコミットメッセージを1行で提案してください。"
```

気に入ったメッセージが返ってきたらそのまま使う。微妙だったら「もっと具体的に」「動詞から始めて」と追加で指示して調整できる。

これもバイブコーディングの一部。AIに任せながら、最終的に採用するかどうかは自分で判断する。

## 7-9. LocalStorage でデータを保存する

現状のアプリはページをリロードするとタスクが消える。LocalStorage を使うことで、ブラウザにデータを保存してリロード後も残るようにできる。

### LocalStorage とは

ブラウザが持つ簡易的なデータ保存の仕組み。サーバーなしで、ユーザーのブラウザにデータを永続的に保存できる。

### spec.md に追記する

まず仕様書を更新してから実装依頼する。

```markdown
## 追加機能
- タスクの追加・完了・削除のたびに LocalStorage に保存する
- ページを読み込んだとき、LocalStorage からタスクを復元して表示する
```

### Gemini に実装させる

```bash
gemini -p "spec.md の追加機能を実装してください。既存のコードは変更せず、LocalStorage の読み書き処理だけ追加してください。"
```

### 生成されたコードで確認すること

LocalStorage の操作には以下の3つが使われているはずなので、それぞれ何をしているか Gemini に聞いて確認する。

```javascript
localStorage.setItem('key', value)  // 保存
localStorage.getItem('key')         // 読み込み
localStorage.removeItem('key')      // 削除
```

確認できたらブラウザでリロードしてタスクが残ることを検証し、コミットする。

```bash
git add script.js
git commit -m "LocalStorageでタスクを永続化"
```

## 7-10. 改善ループ（バグ修正・機能追加をAIと一緒に行う）

### バグを見つけたとき

問題を再現する手順をそのままAIに伝える。

```
以下の手順で操作するとバグが発生します。

手順：
1. タスクを1つ追加する
2. 完了ボタンを押す
3. 削除ボタンを押す

期待する動作：タスクがリストから消える
実際の動作：タスクが消えない

script.js の deleteTask 関数に問題がある可能性があります。確認して修正してください。
```

### 機能を追加したいとき

追加したい機能を `spec.md` に追記してから依頼する。

```bash
# spec.md に追記してから
gemini --context ./spec.md --context ./script.js \
"spec.mdに追加した機能を実装してください。既存のコードは変更しないでください。"
```

### 改善サイクルを止めないコツ

- 1回の依頼で大きく変えすぎない
- 変更するたびにブラウザで確認する
- 動作確認できたらこまめにコミットする
- わからないコードはそのまま使わず、必ずAIに説明させる
