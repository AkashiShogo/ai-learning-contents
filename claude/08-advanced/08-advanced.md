> 実際の開発は「動いた」で終わらない。公開して、壊れていないことを確かめる仕組みを作り、コードを育てていく。この章では完成したTo-Doアプリを題材に、「完成後の開発」を一通り体験する。ここでも主役はAIとの対話だ。

## 8-1. 「完成後」こそ開発の本番

業務の開発では、コードを書く時間より「すでにあるコードを変更する時間」の方が長い。そして変更には常に「壊すリスク」がつきまとう。

この章で作る仕組みは、すべて「安心して変更し続けるため」のもの。

| 仕組み | 何のためか |
|---|---|
| デプロイ | 作ったものを実際に使える状態にする |
| テスト | 変更しても壊れていないことを機械的に確かめる |
| リファクタリング | 変更しやすいコードの形に整える |
| CI | テストを「忘れずに必ず」実行させる |

AIで開発のスピードが上がるほど、変更の回数も増える。**変更が速いチームほど、壊れていないことを確かめる仕組みが必要になる。** エージェント型ツールでAIが直接コードを変えるなら、なおさらだ。テストは「AIの変更が壊していないか」を機械的に見張る、もう一人のレビュアーになる。

## 8-2. GitHub Pages でデプロイする

### GitHub Pages とは

GitHubが提供する静的サイトのホスティング機能。リポジトリに置いたHTML / CSS / JSを、そのままWebサイトとして公開できる。無料で、今回のようなビルド不要のアプリには最適。

> **用語｜ホスティング：** Webサイトのファイルをサーバーに置いて、インターネットからアクセスできるようにすること。

### 公開手順

1. GitHubの `todo-app` リポジトリページを開く
2. `Settings > Pages` を開く
3. 「Source」で `Deploy from a branch` を選び、Branch に `main` / `/(root)` を指定して保存
4. 数分待つと `https://ユーザー名.github.io/todo-app/` で公開される

<!-- TODO: screenshot - GitHub Pages 設定画面 -->

> **公開前に必ず確認：** パブリックに公開される。第2章で学んだ「コミットしてはいけないもの」が含まれていないか、公開する前にもう一度リポジトリ全体を見直す。リポジトリ全体の確認はAIにも手伝わせられる：`> このリポジトリに機密情報や公開すべきでないファイルが含まれていないか確認してください`

### 詰まったらAIに聞く

設定画面のUIは変わることがある。表示が手順と違ったら、自分で止まらずに聞く。

```
> GitHub Pages でリポジトリのmainブランチを公開する手順を教えてください。
> Settings画面には [実際に見えている表示] とあります。
```

### デプロイ後の確認

公開URLをスマホでも開いてみる。PCのブラウザと表示が違う・操作しにくい箇所があれば、それが次の改善ネタになる。

## 8-3. テストとは何か・AIにテストを書かせる

### テストコードとは

「コードが期待通りに動くことを確認するコード」。人間が手でポチポチ確認する代わりに、機械に確認させる。

```
手動確認：タスクを追加して、画面を見て、消して、リロードして…（毎回5分）
テスト：コマンド1つで全部確認（毎回数秒）
```

変更のたびに手動で全機能を確認するのは現実的じゃない。だからテストを書く。

### 何をテストするか

今回のアプリで言えば、たとえば：

- 空文字のタスクは追加されないこと
- タスクを追加するとリストの件数が1増えること
- 削除すると件数が1減ること

**「仕様書に書いたこと」がそのままテスト項目になる。** 第7章で `spec.md` を書いたのはここでも効いてくる。

### AIにテストを書かせる前に：テストしやすい形を知る

実は、第7章のコードはこのままだとテストしにくい。DOM操作（画面の書き換え）とロジック（データの処理）が混ざっているからだ。

```javascript
// テストしにくい：ロジックとDOM操作が混ざっている
function addTask() {
  const input = document.getElementById('taskInput');
  if (input.value === '') return;
  const li = document.createElement('li');
  li.textContent = input.value;
  list.appendChild(li);
}
```

これを「純粋なロジック」と「画面への反映」に分けると、ロジック部分だけを機械的にテストできるようになる。

```javascript
// テストしやすい：ロジックだけの関数（DOMに触らない）
function canAddTask(text) {
  return text.trim() !== '';
}

function addTaskToArray(tasks, text) {
  return [...tasks, { text: text, done: false }];
}
```

この「分ける」作業がリファクタリング。次のセクションで実際にやる。

## 8-4. テストしやすいコードへのリファクタリング

### リファクタリングとは

**挙動を変えずに、コードの構造だけを整理すること。** 機能追加でもバグ修正でもない。「同じ動きをする、もっと良い形」に変える。

### 計画を出させてから、AIにリファクタリングさせる

リファクタリングはAIが得意な作業。ただし変更範囲が広くなりがちなので、第5章で学んだ **「計画 → 承認 → 実行」** の進め方が活きる。

```
> script.js をリファクタリングしたいです。
>
> 目的：ロジックをテストできるようにする
> 方針：
> - データ処理のロジック（タスクの追加・削除・完了の判定）を、DOMに触らない純粋な関数として分離する
> - DOM操作の関数とロジックの関数を分ける
> - 挙動は一切変えない
>
> まず分離する計画だけを提示してください。コードはまだ変更しないでください。
```

計画を読んで納得できたら実行させる。

```
> その計画で進めてください。
```

### リファクタリング後の必須確認

```
□ git diff で変更範囲が計画通りか
□ ブラウザで全機能が今まで通り動くか（追加・完了・削除・リロード）
□ 分離されたロジック関数が本当にDOMに触っていないか（document を参照していないか）
□ 自分で各関数の役割を説明できるか
```

確認できたらコミット。**「リファクタリング」と「機能変更」を同じコミットに混ぜない** のが鉄則。後から履歴を見たときに、挙動の変更点を追えなくなるからだ。

```bash
git add script.js
git commit -m "ロジックとDOM操作を分離するリファクタリング"
```

### テストを書かせる

ロジックが分離できたら、テストをAIに書かせる。Node.js には標準のテスト機能（`node --test`）があるので、追加インストールなしで始められる。

```
> 分離したロジック関数のテストを書いてください。
>
> 条件：
> - Node.js の標準テストランナー（node --test）を使う
> - テストファイルは script.test.js とする
> - 空文字のタスクが追加されないこと・追加で件数が増えること・削除で件数が減ることをテストする
> - テストが通ることを確認してから完了を報告してください
```

最後の1行がエージェント型ならでは。Claude Code は**テストを書いた後、自分で実行して、失敗したら直してから報告する**ところまでできる。

> ロジック関数をテストから読み込むために、`module.exports` や `export` の追加など、ファイル構成の調整が必要になることがある。Claude Code が必要な調整を提案してくるので、内容を理解した上で許可する。

### 自分でも実行して確認する

AIが「テストは通りました」と報告してきても、自分の手で再実行する。

```bash
node --test
```

```
✔ 空文字のタスクは追加されない
✔ タスクを追加すると件数が1増える
✔ タスクを削除すると件数が1減る
```

全部通ったらコミット。**今後は変更のたびにこのコマンド1つで「壊れていないか」を確認できる。**

### テストもレビュー対象

AIが書いたテストには「常に通ってしまう意味のないテスト」が混じることがある。

```javascript
// 意味のないテストの例：何もテストしていない
test('タスクを追加できる', () => {
  assert.ok(true);
});
```

テストコードも読む。**「このテストはどんなバグを検出できるか？」** と自問して、答えられないテストは作り直させる。

## 8-5. GitHub Actions で CI を組む

### CI とは

CI（継続的インテグレーション）は、**コードをプッシュするたびにテストやチェックを自動実行する仕組み**。

テストは「実行し忘れる」ものだ。CIを組むと、忘れても GitHub が勝手に実行してくれる。テストが落ちていればプッシュ後に通知され、壊れたコードに早く気づける。

### GitHub Actions の仕組み

リポジトリの `.github/workflows/` フォルダにYAMLファイルを置くと、GitHubがそれを読んで自動実行する。

> **用語｜YAML：** 設定を書くためのテキスト形式。インデント（字下げ）で構造を表す。

### AIにワークフローを書かせる

```
> GitHub Actions のワークフローファイルを作ってください。
>
> 条件：
> - mainブランチへのプッシュのたびに実行する
> - Node.js をセットアップして node --test でテストを実行する
> - ファイルは .github/workflows/test.yml とする
```

生成されたYAMLの例（おおよそこの形になる）：

```yaml
name: test

on:
  push:
    branches: [main]

jobs:
  test:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-node@v4
        with:
          node-version: 20
      - run: node --test
```

YAMLの各行が何をしているか、わからない行はAIに聞く。

```
> .github/workflows/test.yml の各行が何をしているか説明してください
```

### 動作確認

```bash
git add .github/workflows/test.yml
git commit -m "CIワークフローを追加"
git push origin main
```

GitHubリポジトリの `Actions` タブを開くと、ワークフローが実行されているのが見える。緑のチェックがついたら成功。

### わざと壊して、CIに検出させる

CIの価値は「壊れたときに気づけること」。一度わざと体験しておく。

1. `script.js` のロジック関数を1箇所わざと壊す（例：空文字チェックの条件を逆にする）。これは自分の手で編集する
2. コミットしてプッシュする
3. Actions タブで ❌ になることを確認する
4. 元に戻してプッシュし、✅ に戻ることを確認する

**「CIが落ちる → 原因を直す → 緑に戻る」** の流れを一度体験しておくと、業務でCIに遭遇したときに慌てなくなる。

## 8-6. ここから先の改善ループ

デプロイ・テスト・CIが揃った状態は、業務の開発環境の縮小版だ。ここからは自分で改善ループを回せる。

```
機能を思いつく → spec.md に追記 → AIと実装 → テスト追加 → node --test が緑 → プッシュ → CI が緑 → 公開に反映
```

### 機能追加のアイデア例

- タスクの編集機能
- 期限と並び替え
- カテゴリ・タグ
- ダークモード
- 完了タスクの一括削除

どれを選んでもいい。**仕様を書く → 小さく実装 → テストで守る** のサイクルさえ守れば、アプリは安全に育っていく。

## やってみよう

### ① GitHub Pages でアプリを公開する

8-2 の手順で公開し、公開URLをスマホでも開いて動作確認する。公開前のリポジトリ確認も忘れずに。

---

### ② ロジックを分離するリファクタリングを実施する

8-4 の手順で「計画 → 承認 → 実行」の流れでリファクタリングし、「リファクタリング前後で挙動が同じ」ことをブラウザで確認してからコミットする。

---

### ③ テストを書かせて、自分でも実行する

AIにテストを書かせ、自分の手で `node --test` を実行して全テストが通ることを確認する。

その後、**テストコードを1行ずつ読んで**、各テストが「どんなバグを検出できるか」を説明できるようにする。説明できないテストはAIに質問する。

---

### ④ CI を組んで、わざと落とす

8-5 の手順でワークフローを作り、プッシュしてActionsが動くことを確認する。

その後、わざとコードを壊してCIが落ちることを確認し、直して緑に戻す。

---

### ⑤ 機能をひとつ追加して、フルサイクルを回す

8-6 のアイデアから1つ選び、以下のフルサイクルを自力で回す。

```
spec.md に仕様を追記
→ AIに実装させる（小さく分けて・計画を先に）
→ diff確認・動作確認・コードレビュー
→ テストを追加
→ node --test が通ることを確認
→ コミット & プッシュ
→ CI が緑になることを確認
→ 公開URLで動くことを確認
```

これが一人で回せたら、このコンテンツのゴールは達成だ。
