# 第7章：To-Doリストアプリを作る（実践・Claude トラック）

**目安時間：8時間**

> **前提：** 第3〜6章を終えていること（セットアップ・AIハンドリング・バイブコーディングのサイクルを体験済み）

> これまでの章で学んだことをすべて使う。AIに指示を出し、変更をレビューし、GitHubで管理しながら、動くアプリを完成させる。ただし、アプリの完成はゴールではない。ここで身につけたサイクルを、この先も自分で回し続けられるようになることが本当のゴールだ。

---

## この章で学ぶこと

- 要件定義からコード完成までの一連の流れを体験する
- `CLAUDE.md` と仕様ファイルを使ったプロジェクト管理を実践する
- AIの変更を diff でレビューして問題を見つける
- コミット前チェックを習慣として身につける
- バグ修正・機能追加の改善ループを回す

## 7-1. 作るものを決める（要件定義をClaudeと一緒にやる）

### なぜ要件定義から始めるのか

「とりあえず作り始めて、後から直す」は、最初の方向性が間違っているとやり直しが大きくなる。AIに依頼する前に「何を作るか」を整理することで、ブレのない開発ができる。

エージェント型ツールでは特に重要だ。**仕様が曖昧なまま依頼すると、AIは曖昧なまま実装まで進んでしまう。**

### Claude と一緒に要件定義をする

Claude Code を起動して、以下のプロンプトから始める。

```
> To-Doリストアプリの要件定義を手伝ってください。
>
> 前提：
> - HTML / CSS / Vanilla JS のみで実装
> - 外部ライブラリは使わない
> - 初心者向けの学習目的なので、シンプルな構成にする
>
> まず、最低限必要な機能のリストを提案してください。まだファイルは作らないでください。
```

Claude が提案してきた機能を見て、「これは必要」「これはいらない」を自分で判断する。最終的な機能リストを自分で決める。

### `spec.md` に書き出させる

仕様が固まったら、会話の内容をファイルにまとめさせる。

```
> 今決めた仕様を spec.md にまとめてください。
> 機能一覧はチェックボックス形式で書いてください。
```

生成された `spec.md` を**必ず自分で読んで**、決めた通りになっているか確認する。おおよそ次のような内容になっていればいい。

```markdown
# To-Doリストアプリ 仕様書

## 機能一覧
- [ ] テキスト入力欄にタスクを入力できる
- [ ] 「追加」ボタンを押すとリストにタスクが追加される
- [ ] 入力が空の場合は追加されない
- [ ] 各タスクに「完了」ボタンがあり、押すと取り消し線がつく
- [ ] 各タスクに「削除」ボタンがあり、押すとリストから消える

## 制約
- HTML / CSS / Vanilla JS のみ
- 外部ライブラリ・フレームワーク不使用
- ファイルは index.html・style.css・script.js の3ファイル構成

## 対象ブラウザ
- Chrome 最新版
```

## 7-2. GitHubリポジトリを作り `.gitignore` を設定する

### リポジトリの準備

第3章で作った `todo-app` リポジトリをクローンして、その中で Claude Code を起動する。

```bash
git clone git@github.com:自分のユーザー名/todo-app.git
cd todo-app
claude
```

> 7-1 を別フォルダでやった場合は、`spec.md` を `todo-app` フォルダに移してから進める。

### 初期設定をAIにやらせてみる

ここまでの章では手で作ってきたファイルを、今回はAIに任せてみる。

```
> このプロジェクトの初期設定をしてください。
> - .gitignore を作る（.env・OSの自動生成ファイルを除外）
> - index.html・style.css・script.js の空ファイルを作る
> 中身の実装はまだしないでください。
```

許可プロンプトで操作内容を確認しながら進め、終わったら自分で確認する。

```bash
ls -a
cat .gitignore
```

### 最初のコミットをする

```bash
git add .gitignore spec.md
git commit -m "プロジェクトの初期設定"
git push origin main
```

> コミットはAIに任せることもできるが、**この章では自分の手でコミットする**。何をリポジトリに記録するかの判断を体に入れるのが先。AIにコミットさせる練習は 7-8 でやる。

## 7-3. プロジェクトの `CLAUDE.md` を書く

このプロジェクトで Claude Code を使うときに常に読み込まれる命令書を作る。第5章で学んだ内容の実践。

```markdown
# プロジェクト概要
To-Doリストアプリの学習プロジェクト。

## 技術スタック
- HTML5
- CSS3
- Vanilla JavaScript（ES6以降）

## ファイル構成
- index.html：HTMLの構造
- style.css：スタイル
- script.js：機能の実装
- spec.md：仕様書（実装はこれに従う）

## コーディングルール
- 日本語でコメントを書く
- `var` は使わない。`const` と `let` を使う
- 外部ライブラリは使わない
- ユーザー入力の表示には textContent を使う（innerHTML は使わない）

## 作業ルール
- 実装は spec.md の仕様に従う
- 依頼されていないファイルは変更しない
- 変更したら、何をどう変えたかを箇条書きで報告する
- 機能の追加は1つずつ。複数機能を一度に実装しない
```

書けたらコミットしておく。

```bash
git add CLAUDE.md
git commit -m "CLAUDE.mdを追加"
```

## 7-4. HTMLの骨格を作る

### 実装を依頼する

```
> @spec.md の仕様に基づいて、index.html にHTMLの骨格を実装してください。
> CSSとJSの実装はまだ不要です（ファイルの読み込みタグだけ入れてください）。
```

### diff と実物を確認する

実装が終わったら、いつもの基本動作。

```bash
git diff
```

ブラウザで `index.html` を開いて確認するポイント：

- 入力欄があるか
- 追加ボタンがあるか
- リストが表示される場所があるか
- `id` や `class` が付いているか（JSで参照するために必要）

問題があれば修正指示を出す。

```
> 生成されたHTMLを確認しました。
> 以下の点を修正してください：
> - [具体的な問題点]
```

### コミットする

HTMLが想定通りの構造になったらコミット。

```bash
git add index.html
git commit -m "HTMLの骨格を追加"
```

## 7-5. CSSでスタイルを整える

### CSSを依頼する

```
> index.html に合わせて style.css を実装してください。
> シンプルで見やすいデザインにしてください。
> ボタンはクリックしやすいサイズにしてください。
```

### 確認ポイント

- ブラウザで見て違和感がないか
- ボタンが押しやすいサイズか
- 全体的に読みやすいか

デザインの好みは人によって違うので、気に入らない部分は遠慮なく修正を依頼する。

```
> ボタンの色を青にしてください。
> 文字サイズをもう少し大きくしてください。
```

### コミットする

```bash
git add style.css
git commit -m "CSSのスタイルを追加"
```

## 7-6. JavaScriptで機能を実装する

### 機能を1つずつ実装する

`CLAUDE.md` に「機能の追加は1つずつ」と書いた。それでも依頼側も1つずつ頼むのが基本。

**ステップ1：タスクの追加機能**

```
> spec.md の仕様に基づいて、script.js にタスクの追加機能を実装してください。
> 実装する機能：入力欄のテキストをリストに追加する・追加後に入力欄を空にする・入力が空の場合は追加しない
> 完了・削除機能はまだ実装しないでください。
```

diff を確認 → ブラウザで動作確認 → コミット。

```bash
git diff
# ブラウザで動作確認
git add script.js
git commit -m "タスク追加機能を実装"
```

**ステップ2：完了機能**

```
> タスクの完了機能を追加してください。
> 完了ボタンを押したら取り消し線がつく機能です。
> 既存の追加機能のコードは変更しないでください。
```

diff で「追加機能のコードが変わっていないこと」も確認してからコミット。

**ステップ3：削除機能**

同じように依頼して、diff確認・動作確認・コミット。

### 詰まったときは

エラーが出たら、ブラウザの開発者ツール（`F12` または右クリック→検証）のConsoleタブを開いて、エラーメッセージをそのまま貼り付ける。

```
> ブラウザのコンソールに以下のエラーが出ています。原因を調べて修正してください。
> [エラーメッセージ]
```

Claude Code が原因を調べて修正したら、また diff 確認から。サイクルは変わらない。

## 7-7. 生成されたコードを通しでレビューする

### コード全体を確認する

すべての機能が実装できたら、コード全体を通してレビューする。まず自分で `script.js` を上から読む。説明できない行があれば聞く。

次にAIにもレビューさせる。

```
> script.js 全体をレビューしてください。コードはまだ変更しないでください。
> 以下の観点で確認してください：
> 1. セキュリティ上の問題（XSSなど）
> 2. バグや意図しない動作の可能性
> 3. 不要なコードや重複した処理
```

「変更しないでください」を付けるのがポイント。**レビューと修正を分ける**ことで、指摘の内容を自分で取捨選択してから直させることができる。

### リスク観点でのチェック（第2章の復習）

- 機密情報が含まれていないか
- ユーザーの入力をそのままHTMLに書き出していないか（XSS）

#### XSSの例

```javascript
// 危険な書き方（ユーザー入力をそのまま innerHTML に入れる）
listItem.innerHTML = input.value;

// 安全な書き方（textContent を使う）
listItem.textContent = input.value;
```

`CLAUDE.md` に「textContent を使う」と書いたが、守られているかは別の話。実物のコードで確認したら、必ず理由も聞く。

```
> このコードで textContent を使っていますが、innerHTML ではなく textContent を使う理由を教えてください。
> セキュリティの観点からも説明してください。
```

「動いたから OK」ではなく、**なぜその書き方が選ばれたか**を理解することが、第2章で学んだリスク管理の実践になる。

レビュー結果で問題が出たら、採用する指摘を選んで修正指示を出す。

## 7-8. コミット前チェック

コードをプッシュする前に必ず確認する。

### チェックリスト

```
□ .env などの機密ファイルが .gitignore に含まれているか
□ コード内にAPIキー・パスワードなどが直書きされていないか
□ git status で意図しないファイルがステージングされていないか
□ git diff --staged でコミット内容に依頼外の変更が混じっていないか
□ ブラウザで全機能が動くことを確認したか
```

### コミットメッセージのルール

後から見てわかるメッセージを書く。

```bash
# 悪い例
git commit -m "修正"
git commit -m "update"

# 良い例
git commit -m "タスク削除ボタンを押したときのバグを修正"
git commit -m "CSSのボタンデザインを変更"
```

### Claude Code にコミットさせてみる

Claude Code は git 操作もできる。ここまで自分の手でコミットしてきたから、一度任せてみよう。

```
> 今の変更内容を確認して、コミットしてください。
> コミットメッセージは日本語で簡潔にお願いします。
```

Claude Code は `git status` と `git diff` を確認し、メッセージを考えてコミットする。許可プロンプトで **何がコミットされるか・どんなメッセージか** を確認してから許可する。

終わったら `git log --oneline` で結果を確認する。任せても、**チェックリストの責任が人間にあることは変わらない。**

> `git push` は自動許可にしない（第5章で決めたルール）。プッシュは影響が外に出る操作なので、内容を確認した上で自分の判断で実行する。

## 7-9. LocalStorage でデータを保存する

現状のアプリはページをリロードするとタスクが消える。LocalStorage を使うことで、ブラウザにデータを保存してリロード後も残るようにできる。

### LocalStorage とは

ブラウザが持つ簡易的なデータ保存の仕組み。サーバーなしで、ユーザーのブラウザにデータを永続的に保存できる。

### spec.md に追記する

まず仕様書を更新してから実装依頼する。これも自分で書かずに任せてみる。

```
> spec.md に以下の追加機能を追記してください。
> - タスクの追加・完了・削除のたびに LocalStorage に保存する
> - ページを読み込んだとき、LocalStorage からタスクを復元して表示する
```

追記内容を確認したら実装を依頼する。

```
> spec.md の追加機能を実装してください。
> 既存の機能の動作は変えず、LocalStorage の読み書き処理だけ追加してください。
```

### 生成されたコードで確認すること

LocalStorage の操作には以下の3つが使われているはずなので、それぞれ何をしているか Claude に聞いて確認する。

```javascript
localStorage.setItem('key', value)  // 保存
localStorage.getItem('key')         // 読み込み
localStorage.removeItem('key')      // 削除
```

データがJSON形式で保存される理由（`JSON.stringify` / `JSON.parse` が出てくるはず）も聞いておくと理解が深まる。

確認できたらブラウザでリロードしてタスクが残ることを検証し、コミットする。

```bash
git add script.js spec.md
git commit -m "LocalStorageでタスクを永続化"
```

## 7-10. 改善ループ（バグ修正・機能追加をAIと一緒に行う）

### バグを見つけたとき

問題を再現する手順をそのままAIに伝える。

```
> 以下の手順で操作するとバグが発生します。
>
> 手順：
> 1. タスクを1つ追加する
> 2. 完了ボタンを押す
> 3. 削除ボタンを押す
>
> 期待する動作：タスクがリストから消える
> 実際の動作：タスクが消えない
>
> 原因を調査して修正してください。
```

Claude Code は関連するコードを自分で読んで原因を探す。修正されたら diff 確認 → 再現手順をなぞって動作確認 → コミット。

### 機能を追加したいとき

追加したい機能を `spec.md` に追記してから依頼する。

```
> spec.md に追記した [機能名] を実装してください。既存の機能の動作は変えないでください。
```

### 改善サイクルを止めないコツ

- 1回の依頼で大きく変えすぎない
- 変更するたびに diff とブラウザで確認する
- 動作確認できたらこまめにコミットする
- わからないコードはそのまま使わず、必ずAIに説明させる
- 会話が長くなって精度が落ちたら `/compact` か `/clear` で仕切り直す（`CLAUDE.md` と `spec.md` があるから、リセットしても前提は失われない）

## やってみよう

### ① Claudeと会話しながら要件定義をして、`spec.md` を作る

---

### ② HTML→CSS→JSの順で、1ステップずつ実装させる

7-4〜7-6 の流れで進める。各ステップで必ず：

1. 許可プロンプトの内容を確認して許可
2. `git diff` で変更内容を確認
3. ブラウザで動作確認
4. コミット

---

### ③ 動かない・意図と違う箇所を自分でリストアップして修正指示を出す

全機能を実装したら、すべての機能を手動で操作して確認する。

問題を見つけたら：
1. 何が問題かを1文で書く
2. 修正指示の形式に整える
3. Claude Code に送って修正させる
4. diff で修正範囲を確認する

---

### ④ 完成したコードをレビューしてからGitHubにプッシュする

7-7・7-8 のチェックをすべて実施した上で、最終コミット＆プッシュをする。

```bash
git status   # 意図しないファイルが含まれていないか確認
git add .
git diff --staged   # コミット内容を最終確認
git commit -m "To-Doリストアプリの初期実装を完了"
git push origin main
```

GitHubのリポジトリページを開いて、コードが反映されていることを確認する。

---

### ⑤ LocalStorage を追加してリロード後もタスクが残るようにする

7-9 の手順に従って LocalStorage を実装する。

実装後に以下を確認する。
- タスクを追加 → リロード → タスクが残っているか
- タスクを削除 → リロード → タスクが消えているか
- 完了にしたタスク → リロード → 取り消し線が残っているか
