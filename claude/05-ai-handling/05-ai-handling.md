# 第5章：AIをハンドリングする（Claude トラック）

**目安時間：8時間**

> **前提：** 第3章のセットアップが完了していること（Claude Code が使える状態）

> 「プロンプトを投げてコードをもらう」だけでは、AIの力を半分も使えていない。この章では、AIを意図通りに動かすための設計・制約・コンテキストの渡し方、そしてエージェント型ツールならではの「権限の管理」を学ぶ。

---

## この章で学ぶこと

- プロンプトの構造を理解して、意図通りの回答を引き出せるようにする
- `CLAUDE.md` でAIの振る舞いをコントロールする
- Claude Code にコンテキストを渡す・読ませる方法を覚える
- 許可設定（パーミッション）を自分で管理できるようになる
- AIが間違えやすいパターンを知っておく

## 5-1. プロンプトの構造

プロンプトは「AIへの依頼文」。書き方によって回答の質が大きく変わる。

### 4つの要素

良いプロンプトには以下の要素が含まれている。

| 要素 | 役割 | 例 |
|---|---|---|
| **役割（Role）** | AIにどんな立場で答えさせるか | 「あなたは経験豊富なフロントエンドエンジニアです」 |
| **指示（Instruction）** | 何をしてほしいか | 「以下のコードをレビューしてください」 |
| **制約（Constraint）** | 答え方の条件・制限 | 「箇条書きで3点以内にまとめてください」 |
| **文脈（Context）** | 背景情報・必要な前提 | 「対象はHTML/CSS/JSのみのプロジェクトです」 |

### 要素を組み合わせた例

**悪い例：**
```
コードレビューして
```

**良い例：**
```
あなたは経験豊富なフロントエンドエンジニアです。
script.js をレビューして、問題点を指摘してください。

条件：
- セキュリティ上の問題を最優先で指摘する
- パフォーマンスの問題があれば次に指摘する
- 指摘は箇条書きで、問題点と改善案をセットで書く
```

### エージェント型ツールでは「作業範囲の指定」も要素になる

Claude Code はファイルの変更まで行うため、**「どこまでやっていいか」** を伝えることが重要になる。

```
# 範囲が曖昧（AIが「ついでの修正」までやるかもしれない）
このバグを直して

# 範囲が明確
addTask関数のバグを直してください。
他の関数・他のファイルは変更しないでください。
```

### 要素はすべて必要なわけではない

シンプルな質問は短くていい。要素を詰め込むほど良いわけではなく、**目的に応じて必要な要素だけ入れる** のがコツ。

## 5-2. `CLAUDE.md` でガードレールを設定する

### 常に有効な指示を置く場所

通常のプロンプトが「今この瞬間の依頼」なら、**`CLAUDE.md` は「このプロジェクトで常に守るべきルール」を書いておくファイル**。プロジェクトのルートに置くと、そのフォルダで Claude Code を起動するたびに自動で読み込まれる。

毎回同じ前提を説明しなくていい、プロジェクト専用の命令書だ。

> Web UI（claude.ai）にも「Project」機能でカスタム指示を設定できる仕組みがある。考え方は同じで、「会話のたびに繰り返し書きたくなる前提」を一箇所に固定するもの。

### `CLAUDE.md` の中身の例

```bash
# プロジェクトのルートに CLAUDE.md を作る
touch CLAUDE.md
```

```markdown
# プロジェクトの概要
このプロジェクトはTo-Doリストアプリです。
HTML / CSS / Vanilla JS で実装します。

# 回答の形式
- 日本語で回答する
- コードブロックには必ず言語タグをつける（```html, ```css, ```js）
- 変更を加えたときは、何をどう変えたかを箇条書きで報告する

# 制約
- フレームワーク（React, Vue など）は使わない
- 外部ライブラリは使わない
- コードには日本語のコメントを入れる
- 依頼されていないファイルは変更しない
```

### `/init` で自動生成させる

既にコードがあるプロジェクトでは、`/init` コマンドを使うと Claude Code がプロジェクトを調べて `CLAUDE.md` の下書きを作ってくれる。

```
> /init
```

生成された内容を読んで、自分のルールを追記・修正して使う。**生成させて終わりにせず、必ず自分で読んで調整する**——これも「AIの出力をレビューする」practice のひとつ。

### 設計のポイント

- **具体的に書く**：「わかりやすく答えて」より「箇条書きで3点以内にまとめて」の方が効果がある
- **矛盾を作らない**：指示同士が矛盾していると、AIが混乱した回答を返す
- **短く保つ**：長くなりすぎると守られない指示が出てくる
- **「やってはいけないこと」も書く**：エージェント型では「依頼されていないファイルは変更しない」のような禁止事項が効く

## 5-3. コンテキストの渡し方

AIは「与えられた情報」をもとに回答を作る。前提情報を渡さずに質問すると、的外れな回答が返ってくることがある。

### 何をコンテキストとして渡すか

| 渡すもの | 効果 |
|---|---|
| 仕様書・要件定義 | 何を作るかをAIが理解できる |
| 既存のコード | 「このコードに合わせて」という修正依頼ができる |
| エラーメッセージ | 原因の特定精度が上がる |
| プロジェクトの制約 | フレームワーク・バージョンなどの条件を踏まえた回答になる |
| これまでの会話内容 | 前提を共有したまま話を続けられる |

### Claude Code は自分でファイルを読みに行く

チャット型AIとの大きな違いがここ。Claude Code に「script.js のバグを直して」と頼むと、**自分で script.js を開いて読む**。ファイルの中身をコピペして渡す必要は基本的にない。

明示的に特定のファイルを見せたいときは、`@` でファイルを指定できる。

```
> @spec.md の仕様に基づいて、HTMLの構造を提案してください
```

### コンテキストを渡すときの注意

- **第2章で学んだ通り、機密情報は渡さない**。Claude Code は自分でファイルを読むため、**プロジェクトフォルダに機密ファイルを置かないこと**が前提になる
- コンテキストが多すぎると重要な部分が埋もれる。「このファイルだけ見て」と範囲を絞る指示も有効
- 「関係ありそうだから全部読ませる」より「この作業に必要なものを指定する」

## 5-4. 会話とコンテキストの管理

### 会話が長くなると精度が落ちる

第1章で学んだコンテキストウィンドウの話がここで効いてくる。会話が長くなると、AIは古いやり取りの詳細を失い、最初に伝えた前提を守れなくなることがある。

Claude Code には会話を管理するコマンドがある。

| コマンド | 役割 | 使いどころ |
|---|---|---|
| `/clear` | 会話履歴を完全にリセットする | 別の作業に切り替えるとき |
| `/compact` | 会話履歴を要約して圧縮する | 同じ作業を続けたいが履歴が長くなったとき |

### 使い分けの考え方

```
タスクが変わった       → /clear（前の話は不要。まっさらが一番精度が高い）
同じタスクの続き       → /compact（要点は引き継ぎつつ軽くする）
```

`CLAUDE.md` に書いた前提は `/clear` しても消えない（毎回自動で読み込まれる）。**「会話に流すもの」と「CLAUDE.md に固定するもの」を分ける** のが、コンテキスト設計の基本形。

```
CLAUDE.md   → プロジェクト全体の前提・ルール（常に有効）
@ファイル参照 → 今回の作業に必要なファイル
プロンプト   → 今回やってほしいこと
```

## 5-5. 許可設定（パーミッション）を管理する

### なぜ自分で管理するのか

第2章2-7で学んだとおり、エージェント型ツールの許可は「全部許可」でも「全部確認」でもなく、**操作のリスクに応じて段階をつける** のが正解。Claude Code はこれを自分で設定できる。

### 許可の仕組み

Claude Code は、ファイル編集やコマンド実行の前に許可を求めてくる。許可には選択肢がある。

- **今回だけ許可する**
- **このセッション・プロジェクトでは今後も許可する**（同種の操作の確認を省略）
- **拒否する**（理由を伝えて別の方法を取らせることもできる）

「今後も許可する」を選んだ内容は、プロジェクトの `.claude/settings.local.json` などの設定ファイルに記録される。`/permissions` コマンドで現在の許可ルールを確認・編集できる。

```
> /permissions
```

### 最小権限の原則で段階をつける

第2章の表をClaude Codeの運用に落とすと、こうなる。

| 操作 | 設定の目安 |
|---|---|
| ファイルの読み取り・検索 | 自動許可でよい |
| プロジェクト内のファイル編集 | 慣れてきたら自動許可。ただし編集後の `git diff` 確認は省略しない |
| `git add` / `git commit` | 慣れるまでは毎回確認 |
| `rm` などの削除・`git push`・外部への送信 | **常に毎回確認**（自動許可にしない） |

> **設定ファイルの中身も読んでみる：** `.claude/settings.local.json` を開くと、自分が何を許可してきたかが一覧できる。「いつの間にか広く許可しすぎていないか」を時々見直す。この「権限の棚卸し」は業務でAIツールを使うときにそのまま活きる。

### 計画を先に出させる

大きめの作業を頼むときは、いきなり実行させず **計画を先に出させて、人間が承認してから実行に移る** という進め方ができる。

```
> To-Doアプリにタスク編集機能を追加したい。
> まず実装計画だけを提示してください。コードはまだ書かないでください。
```

計画を読んで、方向性が合っていれば「その計画で進めて」と指示する。手戻りが大きく減る。Claude Code にはこの「計画してから実行する」流れを支援するモード（プランモード）もある。`/help` や公式ドキュメントで確認してみよう。

## 5-6. AIの回答を引き出すコツ

### 一度で完璧な回答を求めない

AIとの対話は「一問一答」ではなく「キャッチボール」。最初から完璧な回答を期待するより、何回かやり取りして精度を上げていく方が結果がいい。

```
1回目：大まかな方向性を出させる
2回目：「〇〇の部分をもっと詳しく」「△△は不要」と修正
3回目：「他の選択肢はあるか」「このアプローチの問題点は」と深掘り
```

### 「なぜ？」を聞く

回答が返ってきたら、そのまま使うのではなく「なぜそうするのか」を聞く。

```
> さっきの変更で、[特定の部分] をそうした理由を教えてください。
> 他のアプローチと比べてどちらが良いですか？
```

理由を理解することで、生成されたコードを自分で判断できるようになる。

### 「制約を与えて絞る」か「制約をなくして広げる」か

| 目的 | やること |
|---|---|
| 特定の条件に合う答えが欲しい | 制約を明示する |
| 選択肢を出してほしい | 「3つの方法を提示してください」 |
| 自分の考えを確認したい | 「この方法の問題点を教えてください」 |
| 全体像をつかみたい | 「まず概要だけを説明してください」 |

### 作業を止める・軌道修正する

Claude Code が作業中に「方向が違う」と気づいたら、`Esc` キーで中断して指示し直せる。**間違った方向に走り続けさせない** のもハンドリングの一部。

## 5-7. AIが間違えやすいパターンを知る

AIは間違えることがある。よくある間違いのパターンを知っておくと、出力を見たときに「これは怪しい」と気づけるようになる。

### パターン① 自信満々に間違える（ハルシネーション）

AIは間違っていても断定的に答えることがある。「〇〇です」という表現を信用しすぎない。

**見分け方：** 重要な事実（関数名・APIの仕様・ライブラリのバージョンなど）は公式ドキュメントで確認する。

### パターン② 動かないコードを書く

コードが「見た目は正しそう」でも、実際に動かすとエラーが出ることがある。

**見分け方：** 変更後は必ず自分で実行して確認する。動くまでは完成とみなさない。

### パターン③ 古い情報を使う

学習データのカットオフより後にアップデートされた情報は知らない。

**見分け方：** バージョンに依存する情報（APIの仕様・非推奨になった機能など）は必ずドキュメントで確認する。

### パターン④ 曖昧な指示を都合よく解釈する

指示が曖昧なとき、AIは「もっともらしい解釈」で答える。意図と違う方向に進むことがある。

**見分け方：** 回答が意図と違う場合は、指示の曖昧な部分を特定して言い直す。

### パターン⑤ 頼んでいないことまでやる（エージェント型特有）

Claude Code は「ついでにここも直しておきました」と、依頼範囲の外まで変更することがある。良かれと思っての行動だが、意図しない変更はバグの温床になる。

**見分け方：** 変更後の `git diff` で、**依頼していないファイル・行が変わっていないか** を確認する。

**対策：** `CLAUDE.md` に「依頼されていないファイルは変更しない」と書く。依頼時にも「他は変更しないで」と範囲を区切る。

### パターン⑥ 途中で仕様を忘れる

会話が長くなると、最初に伝えた前提を守れなくなることがある。

**見分け方：** 「最初に〇〇という条件を伝えましたが、それは守られていますか？」と確認する。

**対策：** 守らせたい前提は会話ではなく `CLAUDE.md` に書く。会話が長くなったら `/compact` か `/clear` で仕切り直す。

## 5-8. 詰まったときのチェックリスト

AIの動きが期待と違うとき、闇雲にプロンプトを書き直す前に、上から順に確認する。

```
□ 指示は具体的か（「いい感じに」「ちゃんと」を使っていないか）
□ 作業範囲を区切ったか（「他は変更しないで」）
□ 必要な前提・制約を渡しているか（仕様・バージョン・CLAUDE.md）
□ 1つの依頼に複数の作業を詰め込んでいないか
□ 会話が長くなりすぎていないか（/compact か /clear で仕切り直す）
□ そもそもAIが答えられる質問か（最新情報・社内固有の話ではないか）
```

体感では、期待と違う動きの大半は「指示が曖昧」か「前提が足りない」のどちらか。**AIの性能を疑う前に、自分の渡した情報を疑う。**

## やってみよう

### ① 「役割あり」と「役割なし」で回答を比較する

同じ質問を2通りの形式で送って、回答の違いを比べる。

```
# 役割なし
JavaScriptのエラーハンドリングのベストプラクティスを教えてください。
```

```
# 役割あり
あなたはJavaScriptを10年書いてきたシニアエンジニアです。
初心者のJSエンジニアに向けて、エラーハンドリングのベストプラクティスを教えてください。
注意点と具体的なコード例も含めてください。
```

---

### ② `CLAUDE.md` を作って回答スタイルが変わることを確認する

practice フォルダに `CLAUDE.md` を作り、以下を書いて保存する。

```markdown
# ルール
- 回答は必ず日本語
- コードブロックには言語タグをつける
- 説明は箇条書きを優先する
- 初心者にもわかるように、専門用語には補足説明をつける
- 依頼されていないファイルは変更しない
```

いったん `/exit` で終了して `claude` を再起動し、何か質問してスタイルが変わるか確認する。

---

### ③ 仕様ファイルを読ませて回答させる

以下の内容で `spec.md` を作る。

```markdown
# To-Doリストの仕様

## 機能
- テキストボックスにタスクを入力してボタンを押すと、リストに追加される
- リストの各タスクに「完了」ボタンがあり、押すと取り消し線がつく
- リストの各タスクに「削除」ボタンがあり、押すとリストから消える

## 制約
- HTML / CSS / Vanilla JS のみで実装する
- 外部ライブラリは使わない
```

Claude Code で `@` 参照を使って聞いてみる。

```
> @spec.md の仕様を実現するためのHTMLの構造を提案してください。まだファイルは作らないでください。
```

「まだファイルは作らないでください」の一文があるかないかで動きがどう変わるかも、外して試して比較してみる。

---

### ④ 意図的に曖昧なプロンプトを送って誤解を体験する

以下のような曖昧なプロンプトを送ってみよう。

```
> リストを作ってください
```

どんな「リスト」が返ってくるか（あるいはどんなファイルを作ろうとするか）を確認する。次に、意図を明確にしたプロンプトを送り直して違いを確認する。

---

### ⑤ 許可設定を確認して、自分のルールを決める

`/permissions` で現在の許可ルールを表示する。`.claude/settings.local.json` があれば中身も開いて読む。

そのうえで、5-5 の表を参考に「自分はどの操作を自動許可にして、どの操作を毎回確認にするか」を書き出す。第7章の開発はこのルールで進める。

---

### ⑥ 計画を出させてから実行させる

```
> practice フォルダに、自己紹介ページ（profile.html）を作りたいです。
> まず構成の計画だけ提示してください。ファイルはまだ作らないでください。
```

計画を読んで、1箇所だけ注文をつけて修正させる。納得できたら「その計画で作ってください」と実行させ、結果を確認する。

**計画 → 承認 → 実行 → 確認** の流れを一度通しでやっておく。
