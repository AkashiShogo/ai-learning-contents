## 4-1. 公式ドキュメントはなぜ読むべきか

### ブログ・Qiita・記事との違い

ネットで調べると、ブログや技術記事がたくさん出てくる。読みやすいし、すぐ使えそうな情報が多い。でも、これらには共通の問題がある。

**書かれた時点の情報しか保証されない。**

AIツールのバージョンが変わったり、APIの仕様が変わったりすると、記事の内容が古くなる。古い情報をそのまま試してエラーが出て詰まる、というのはよくある話。

一方、公式ドキュメントは開発元が管理しているため：

- **最新の仕様が書かれている**
- **動作が保証されている手順が書かれている**
- **変更があった場合に更新される**

公式ドキュメントを読む習慣がある人とない人では、詰まったときの解決速度が大きく変わる。

### 「難しそう」という先入観を捨てる

公式ドキュメントは英語が多い、難しい言葉が並んでいる、と感じる人は多い。でも全部読む必要はない。

**目的のページに直接飛んで、必要な部分だけ読む** で十分。わからない単語は Gemini に説明してもらえばいい。

---

## 4-2. Gemini の公式ドキュメントの構造と読み方

### 主要なドキュメントの場所

Gemini 関連のドキュメントは主に2つに分かれている。

| ドキュメント | URL | 対象 |
|---|---|---|
| Google AI for Developers | [https://ai.google.dev/](https://ai.google.dev/) | API・モデル・開発者向け |
| Gemini for Google Workspace | [https://workspace.google.com/products/gemini/](https://workspace.google.com/products/gemini/) | 企業向けサービス・データポリシー |
| Gemini CLI | [https://github.com/google-gemini/gemini-cli](https://github.com/google-gemini/gemini-cli) | CLI ツールの使い方 |

### 読み方のコツ

**①「Getting Started（はじめに）」から読まない**

目次を見て、今知りたいことが書いてあるセクションに直接飛ぶ。

**② ページ内検索を使う**

`Ctrl + F`（macOS は `Command + F`）でページ内検索して、知りたいキーワードを探す。

**③ わからない単語は飛ばさない**

わからない単語が出てきたら Gemini に聞く。「ドキュメントを読むのをやめる」のではなく「その場で解決する」。

## 4-3. Gemini 自身に「Gemini について」聞く

「AIでAIのことを調べる」は一見おかしく聞こえるが、実際にはとても有効な方法。

### なぜ有効なのか

Gemini は自分自身のドキュメントや仕様についての情報も学習している。難しいドキュメントを読む前に、まず Gemini に「わかりやすく説明して」と頼むことで、ドキュメントの理解が速くなる。

ただし注意点が1つ。

> **Gemini の回答には学習データのカットオフがある。** 最新の機能やアップデートについては、正確でない場合がある。Gemini の回答を「理解のための補助」として使い、最終確認は公式ドキュメントで行う。

### 具体的な使い方

ドキュメントを読んでいてわからない部分をコピーして、Gemini に貼り付ける。

```
以下のドキュメントの内容を、エンジニア歴3年未満の人向けにわかりやすく説明してください。

[ドキュメントの文章をここに貼り付ける]
```

または、特定の機能について概要を聞く。

```
Gemini CLI の --context オプションは何をするものですか？
具体的な使用例も教えてください。
```

## 4-4. 情報の鮮度と信頼性を見極める

AIツールの情報を調べるときは、「いつ書かれたか」が重要。

### 鮮度を確認するポイント

| 確認すること | 方法 |
|---|---|
| 記事の公開・更新日 | ページ上部か下部に表示されていることが多い |
| 対象バージョンの明記 | 「Gemini 1.5 Pro 時点」などの記述があるか |
| URLに年号が入っているか | `/2023/` のようなURLは古い可能性がある |

### 信頼性を確認するポイント

| 確認すること | 判断基準 |
|---|---|
| 書いた人・組織 | 公式か、信頼できる個人や組織か |
| 参照元が明記されているか | 「公式ドキュメントによると」などの引用があるか |
| 実際に試した結果があるか | 「やってみたら動いた」という実証があるか |

### Gemini に情報の確認を頼む

```
以下の情報は現在でも正確ですか？また、この情報はいつ頃のものですか？

[確認したい情報を貼り付ける]
```

Gemini も間違えることがあるので、重要な情報は複数の情報源で確認する習慣をつける。

## 4-5. 「わからないこと」を言語化してAIに聞く技術

AIへの質問で最もよくある失敗は、「質問が曖昧すぎる」こと。

### 悪い質問と良い質問の違い

**悪い例：**
```
エラーが出ました。どうすればいいですか？
```

**良い例：**
```
以下のエラーが出ています。
環境：macOS、Node.js v20、Gemini CLI v1.0
実行したコマンド：gemini --context ./spec.md "HTMLを作ってください"
エラーメッセージ：[エラー内容をここに貼り付ける]

このエラーの原因と解決方法を教えてください。
```

### 質問に含めるべき情報

1. **何をしようとしていたか**（目的）
2. **何をしたか**（実行した操作・コマンド）
3. **何が起きたか**（エラーメッセージ・実際の結果）
4. **環境情報**（OS・ツールのバージョンなど）

### わからないことを言語化するコツ

「何がわからないかがわからない」状態のときは、まず以下を Gemini に聞く。

```
[学んでいる内容] について理解できているかを確認したいです。
私の理解が正しいか確認してください。

私の理解：[自分の理解を書く]
```

自分の理解を書き出すことで、どこがずれているかが明確になる。

## やってみよう

### ① Gemini の公式ドキュメントを読む

[Gemini CLI の GitHub リポジトリ](https://github.com/google-gemini/gemini-cli) を開いて、README を読む。

読んだ後、以下を書き出してみよう。
- わかったこと
- わからなかったこと
- もっと知りたいこと

---

### ② ブログ記事と公式ドキュメントを比較する

「Gemini CLI 使い方」で検索して出てきたブログ記事を1つ選ぶ。

その記事に書かれている内容を1つ取り上げて、公式ドキュメントに同じ情報があるか確認する。

- 情報は一致しているか？
- 記事はいつ書かれたものか？

---

### ③ わからないことを1文で書いてGeminiに聞く

この章を読んで、まだよくわからないことを1文で書き出す。

書けたら、4-5 で学んだ「良い質問」の形式に整えて Gemini に送ってみよう。

---

### ④ Gemini の回答に「それはいつの情報か」を聞く

何か Gemini に質問して回答をもらったら、続けてこう聞く。

```
その情報はいつ頃のものですか？最新の状況と変わっている可能性はありますか？
```

回答を見て、情報の鮮度への意識を持つ習慣をつける。
