## 6-1. バイブコーディングとは何か

### 定義

バイブコーディング（Vibe Coding）は、AIとの対話を中心にコードを作っていく開発スタイル。

「自分でコードを書く」のではなく、「AIに作らせて、確認して、修正させる」というサイクルを繰り返す。

### 従来の開発との違い

| 従来の開発 | バイブコーディング |
|---|---|
| 仕様を理解 → 自分で設計 → コードを書く | 仕様をAIに伝える → AIが生成 → 確認・修正を繰り返す |
| コードを書く時間が長い | AIとの対話時間が長い |
| コードを書く力が必要 | コードを読む力・指示を出す力が必要 |
| 詰まると調査に時間がかかる | 詰まったらAIに聞ける |

### 「書けなくても開発できる」は半分正しい

バイブコーディングは「コードを書けなくてもアプリが作れる」という文脈で語られることがある。これは一面では正しいが、落とし穴がある。

**生成されたコードを読めないと：**
- バグがあっても気づかない
- セキュリティ上の問題があっても見逃す
- 意図と違うコードが動いていても判断できない

**バイブコーディングで本当に必要な力は「読む力」。**

AIにコードを書かせながら、「このコードは何をしているか」「意図通りか」を判断できる状態を目指す。

## 6-2. コード生成のサイクル

バイブコーディングの基本サイクルは以下の4ステップ。

```
① 依頼する
　　↓
② 出力を確認する
　　↓
③ 動かして確認する
　　↓
④ 問題があれば修正指示を出す → ① に戻る
```

### ① 依頼する

仕様や条件を整理してAIに渡す。5章で学んだプロンプトの構造を使う。

**ポイント：** 一度に大きな機能を依頼するより、小さく分けて依頼する方がうまくいく。

```
# 悪い例（一度に大きく頼みすぎ）
To-Doリストアプリを作ってください

# 良い例（小さく分けて頼む）
まず、テキスト入力欄と追加ボタンだけのHTMLを作ってください。
CSSとJSはまだ必要ありません。
```

### ② 出力を確認する

コードが返ってきたら、いきなりコピペせずに読む。

確認すること：
- 依頼した内容が実装されているか
- 余計なものが含まれていないか
- 知らない書き方が含まれていないか（わからない部分はAIに聞く）

### ③ 動かして確認する

コードをファイルに貼り付けてブラウザで確認する。見た目が合っているか、機能が動くかを確認する。

動かない・見た目が違う場合は次のステップへ。

### ④ 修正指示を出す

問題を具体的に伝えて修正させる。

```
# 曖昧な修正指示（悪い例）
なんか違います。直してください。

# 具体的な修正指示（良い例）
ボタンを押してもリストに追加されません。
JavaScriptのaddTask関数が呼ばれていないようです。
ボタンのonclick属性が正しく設定されているか確認して修正してください。
```

## 6-3. 生成されたコードを読む視点

生成されたコードを評価するために必要な視点を整理する。

### 視点① 動くかどうか

最初の確認はシンプルに「動くかどうか」。ブラウザで開いて、意図した通りに動作するか確認する。

### 視点② 意図通りかどうか

動いていても、依頼した内容と合っているかは別の話。

- 「ボタンを押したらリストに追加」という依頼に対して、ページをリロードして追加されていないか
- 削除ボタンが「削除」ではなく「完了」の動きをしていないか

### 視点③ 理解できるかどうか

コードの各行が「何をしているか」を説明できるか確認する。

説明できない部分があれば Gemini に聞く。

```
以下のコードについて、各行が何をしているか説明してください。

[コードを貼り付ける]
```

### 視点④ 問題がないかどうか

機能が動いていても、以下の問題が潜んでいることがある。

| 確認ポイント | 内容 |
|---|---|
| セキュリティ | ユーザーの入力をそのままHTMLに書き出していないか（XSS） |

> **用語｜XSS（クロスサイトスクリプティング）：** ユーザーの入力をそのままHTMLに埋め込むことで、悪意あるスクリプトが実行されてしまう脆弱性。`innerHTML` ではなく `textContent` を使うことで防げる。第7章で実際に確認する。
| 不必要な処理 | 使っていない変数・関数が含まれていないか |
| 依頼した制約が守られているか | 「外部ライブラリを使わない」など、System Instructions の制約を守っているか |

## 6-4. よくある生成ミスのパターンと見分け方

### ミス① イベントリスナーが重複する

「ボタンを押すたびに動作が増えていく」という症状が出たら、イベントリスナーが複数回登録されている可能性がある。

```javascript
// 問題のあるパターン
function setup() {
  const btn = document.getElementById('addBtn');
  btn.addEventListener('click', addTask);  // 関数が呼ばれるたびに追加される
}
```

### ミス② `null` チェックがない

要素が見つからない場合にエラーが起きる。

```javascript
// 問題のあるパターン
const input = document.getElementById('taskInput');
const value = input.value;  // input が null の場合にエラー

// 安全なパターン
const input = document.getElementById('taskInput');
if (!input) return;
const value = input.value;
```

### ミス③ HTMLのid名が一致しない

HTMLのidとJSで参照しているidが違う。コードを通しで見たときに気づきやすい。

```html
<!-- HTML -->
<input id="task-input" />
```

```javascript
// JS（idが違う）
const input = document.getElementById('taskInput');  // null になる
```

### ミス④ 古い書き方が混じる

学習データに古いコードが含まれているため、非推奨の書き方が出てくることがある。

```javascript
// 古い書き方
var x = 1;

// 現在の書き方
const x = 1;
let y = 2;
```

見つけたら修正を依頼する。

## 6-5. AIへのフィードバックの仕方

### 修正指示の基本

効果的な修正指示には以下の情報が含まれている。

1. **どこが問題か**（ファイル名・関数名・行番号など）
2. **どうなっているか**（現在の動作・エラーメッセージ）
3. **どうしてほしいか**（期待する動作）

```
addTask関数に問題があります。
現在の動作：ボタンを押すと入力値が追加されますが、入力欄がクリアされません。
期待する動作：タスク追加後に、入力欄を空にしてください。
```

### 部分的な修正を依頼する

コード全体を再生成させると、他の部分が変わってしまうことがある。修正したい部分だけを指定する。

```
以下のコードの addTask 関数のみを修正してください。
他の部分は変更しないでください。

[修正したい関数のコードを貼り付ける]
```

### 理由を聞きながら修正させる

修正後に「なぜそうしたか」を聞くことで、次に同じ問題が起きたときに自分で対処できるようになる。

```
修正してくれた内容について、なぜその方法を選んだか説明してください。
他にも修正方法はありますか？
```

## やってみよう

### ① 「ボタンを押したらアラートを出すHTML」を生成させて動かす

以下のプロンプトを Gemini に送る。

```
ボタンを押したら「Hello!」とアラートを出すHTMLファイルを作ってください。
HTML / CSS / JavaScript をすべて1つのファイルにまとめてください。
```

生成されたコードを `hello.html` という名前で保存して、ブラウザで開いて動作確認する。

---

### ② コードを1行ずつ読んで、わからない部分をGeminiに聞く

生成されたコードを見て、説明できない行があったら Gemini に聞く。

```
以下のコードについて、各行が何をしているか初心者向けに説明してください。

[コードを貼り付ける]
```

すべての行の意味が理解できるまで繰り返す。

---

### ③ 不完全な仕様を渡してどこが欠けるか確認する

以下の曖昧な仕様を渡してみよう。

```
タスクを管理するHTMLを作ってください。
```

出てきたコードを確認して、「自分の想定と違う部分」をリストアップする。その後、仕様を明確にして再依頼し、違いを比べる。

---

### ④ バグを修正指示で直させる一往復を体験する

③ で生成したコードを意図的に確認して、1つ不具合や気になる点を見つける。

見つけたら、6-5 の形式で修正指示を作って送ってみよう。
