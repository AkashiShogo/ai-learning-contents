# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## このリポジトリの目的

エンジニア歴3年未満でAIツールを使ったことがない人向けの学習コンテンツ集。AIとは何かを理解するところから、最終的にTo-Doリストアプリをバイブコーディングで完成させるところまでを段階的に学べる。

**ターゲット読者：**
- エンジニア歴3年未満
- AIツールの使用経験がほぼない
- 「便利そうとは聞くけど何からやればいいかわからない」という状態

**2トラック構成：**

| トラック | 使用ツール |
|---|---|
| Gemini トラック | Gemini Enterprise（Google Workspace）＋ Gemini CLI |
| Claude トラック | Claude（claude.ai）＋ Claude Code |

学習者はどちらか一方を選んで進める。**両トラックのゴール・到達レベルは同一**（To-Doアプリの完成＋第8章のデプロイ・テスト・CI）。

**技術スタック（題材アプリ）：** HTML / CSS / Vanilla JS（ビルド不要・ブラウザだけで動く）

**開発環境（最低限必要なもの）：**
- Visual Studio Code
- ターミナル（macOS: Terminal / zsh、Windows: PowerShell / WSL）
- トラックのAI CLIツール（Gemini CLI または Claude Code）
- Git / GitHub

**最終ゴール：** AIをハンドリングしながらバイブコーディングでTo-Doリストアプリを作り、生成されたコードの正誤を自分で判断できるようになる。第8章でデプロイ・テスト・CIまで体験する

**コンテンツの核心：**
- プロンプトを投げるだけでなく、常時指示（GEMINI.md / CLAUDE.md）・ガードレール・コンテキスト設計でAIを制御する
- バイブコーディング（AIとの対話でコードを作る）のサイクルを身につける
- AIの出力を鵜呑みにせず「正しいか・正しくないか」を判別する視点を育てる
- **公式ドキュメントを読む習慣と、AIを使った情報収集の手段を身につける**（コンテンツ卒業後も自走できる状態にする）
- **AIを使う際のリスクと人間が管理すべき領域を理解する**（情報漏洩・コミットしてはいけないもの・AIに渡してはいけないデータ・エージェント型ツールの権限管理）

## コンテンツ設計の方針

### 章構成と難易度の段階

| 章 | 置き場所 | テーマ | 時間 |
|---|---|---|---|
| 第1章 | common | AIとは何か | 3h |
| 第2章 | common | AIを使う前に知るべきリスク | 4h |
| 第3章 | トラック別 | 開発環境のセットアップ | 6h |
| 第4章 | common | 情報を自分で集める力を身につける | 4h |
| 第5章 | トラック別 | AIをハンドリングする | 8h |
| 第6章 | トラック別 | バイブコーディング入門 | 7h |
| 第7章 | トラック別 | To-Doリストアプリを作る（実践） | 8h |
| 第8章 | トラック別 | アプリを育てる（デプロイ・テスト・CI） | 6h |

**総学習時間：約46時間（1日8時間 × 約6日間）**

読む順番：00 → 01 → 02 →（トラック）03 → 04 →（トラック）05 → 06 → 07 → 08

### 2トラック運用のルール

- **common/ にツール固有の手順を書かない**。ツール名は例示として出してよいが、操作手順・コマンド・固有機能の解説は必ずトラック側に置く
- **両トラックは構成を揃える**。章番号・セクションの並び・演習の意図を対応させ、片方だけ内容を更新したら、もう片方にも同じ趣旨の更新が必要か必ず確認する
- トラック固有の強みは消さない（例：Claude トラックの diff レビュー・許可管理、Gemini トラックのパイプ活用）
- 章をまたぐ参照は「第N章」のように章番号で書く（パスは書かない）

### 各章共通の4パート構成

1. **解説**：概念・仕組み・背景
2. **やってみよう**：実際に手を動かす演習
3. **理解度チェック**：自分の言葉で答える設問（必ずセクション参照「（→ N-N）」をつける）
4. **もっと深く知るために**：公式ドキュメントリンク・AIに聞くサンプルプロンプト・次に調べるキーワード

### 書き方の原則

- **読者は「よくわからない」状態からスタートしている**：専門用語は出たタイミングで必ず説明する
- **Why → What → How の順**：なぜ必要か → 何をするか → どうやるか の順で説明する
- **コードブロックには必ず言語タグをつける**：\`\`\`bash, \`\`\`js など
- **スクリーンショットが必要な箇所はコメントで明示する**：`<!-- TODO: screenshot -->`
- **1ファイル = 1トピック**：詰め込まない。長くなったら分割を検討する
- **シンプルさ優先**：余計な説明は削る。「大切なことだけ伝える」が基準
- **答えを全部渡さない**：調べ方・聞き方を示す。「AIに聞いて確認する」演習を多用する
- **情報収集スキルは独立した章として扱う**：ツールの使い方ではなく「自分でAI情報を集めて評価する力」そのものを第4章に置く

## コンテンツ作成時の注意

- 読者の知識レベルを都度確認しながら書く。「エンジニア3年未満なら知っている」ことと「知らない」ことの線引きを意識する
- 実際に手を動かせるサンプルを優先する。読むだけで終わるコンテンツにしない
- ツールの固有UIは変わりやすい。手順の説明は概念ベースにして、細かいUI操作の記述は最小限にする。変わりやすい箇所には「公式ドキュメントで最新を確認」の導線を置く
- データポリシー・料金・インストール手順など変化の速い情報は、断定を避けて公式ページへの確認を促す書き方にする
