# はじめに：このコンテンツを読む前に

---

## AIはエンジニアの選手生命を脅かすツールである

はっきり言う。

AIは「エンジニアの仕事を楽にするツール」だが、**楽になった時間を怠惰に使うエンジニアを確実に淘汰される**。

コードを生成させて、動いてるっぽいから承認してプッシュする。エラーが出たら貼り付けてまた承認する。そのサイクルを繰り返すだけのエンジニアは、AIが普及すればするほど必要とされなくなる。なぜなら、その仕事はAI単体でほぼ完結しているからだ。

これは脅しではなく、現実の話をしている。

---

## 同時に、強力な武器でもある

使い方次第で、AIはエンジニアの仕事を大きく広げる。

**理解を深める**  
わからないことをその場で聞ける。生成されたコードの理由を即座に確認できる。別の実装方法を比較できる。従来は「とりあえず動けばいい」で止まりがちだった部分を、AIがいれば掘り下げ続けられる。

**試行回数を増やす**  
実装の初稿をAIに出させることで、1人で全部考えるより多くの選択肢を短時間で検討できる。「いいものを作る」ためのサイクルが速くなる。

**得意でない領域をカバーする**  
CSSが苦手なエンジニアがスタイルの初稿を作れる。テスト設計が不慣れでもAIと一緒に考えられる。自分の弱い部分を補いながら、本来集中すべき設計や判断に時間を使える。

**アイデアを広げる**  
「こういうことをしたいが、どんな方法があるか」という問いに対して、AIは複数の選択肢を出してくれる。1人では思いつかなかったアプローチに出会えることもある。

**AIが登場したことで、エンジニアが「できること」の幅は確実に広がった。** この広がりをどう活かすかが、これからのエンジニアとしての成長速度を決める。

---

## このコンテンツで身につけてほしいこと

このコンテンツを終えたとき、「Gemini が使えるようになった」だけで終わりにしないでほしい。

AIは今も進化し続けている。今日覚えた使い方が、半年後には古くなっていることもある。「使えた」という過去の事実は、すぐに陳腐化する。

**必要なのは、変化し続けるAIを自分で学び、使いこなし続けられる力だ。**

新しいツールが出たとき、自分でドキュメントを読んで試せるか。わからないことをAIに聞いて理解を深められるか。生成された出力を批判的に評価できるか。

このコンテンツはその「きっかけ」にすぎない。ここで学ぶのは特定のツールの操作方法ではなく、**自走してAIと向き合い続けるための土台**だ。

| 用語 | 説明 |
|---|---|
| **LLM**（大規模言語モデル） | 大量のテキストを学習し、言葉のパターンから文章を生成するAIシステム。ChatGPT・Gemini・Claude などの基盤技術 |
| **プロンプト** | AIへの入力文・指示文。AIはプロンプトをもとに回答を生成する |

| **RAG** | 外部のデータベースや文書を検索してAIの回答に組み込む技術。最新情報やカットオフ以降の情報を扱える |
| **AIエージェント** | 指示を受けて自律的にタスクを実行するAI。ツールを使ったり複数の処理を組み合わせたりできる |
| **バイブコーディング** | AIとの対話を中心にコードを作り上げていく開発スタイル |

AIは間違える。自信満々に間違える。

それを見抜けるのは、コードを理解しているエンジニアだけだ。

理解していなければ、間違ったコードを本番に送り出す。セキュリティの穴を見逃す。意図しない動作に気づかない。

## 1-1. LLM（大規模言語モデル）とは何か

### なぜ仕組みを知る必要があるのか

AIを使いこなすには、AIが「どうやって動いているか」のおおまかなイメージを持っておくことが重要。

全部理解する必要はない。でも「なんとなく仕組みがわかってる」状態と「全くわからない」状態では、AIとの対話の質が大きく変わる。なぜなら、仕組みを知っていると「なぜAIがこういう答えを返したのか」「どんな頼み方をすれば意図通りに動くか」が推測できるようになるから。

### LLMとは

ChatGPT、Gemini、Claude といった会話できるAIは、**LLM（Large Language Model＝大規模言語モデル）** と呼ばれる技術をベースにしている。

LLMは一言でいうと、**「大量のテキストから言葉のパターンを学んだ予測エンジン」**。

インターネット上の膨大な文章（書籍・ニュース・コード・論文など）を読み込み、「この文脈のあとにはどんな言葉が続くか」を繰り返し学習している。質問を投げると、学習したパターンをもとに「最も自然な続き」を生成して返す。

### LLMが「知っている」ということの意味

ここで重要な点がある。LLMは「知識を持っている」というより、「知識があるように見える文章を生成できる」と理解した方が正確。

- 学習データに含まれていた情報は答えられる
- 学習データの**カットオフ日時以降**の情報は持っていない
- 確信がなくても、もっともらしい文章を生成することがある（これを**ハルシネーション**という）

> **用語｜カットオフ：** AIの学習データの締め切り日時。例えばカットオフが2024年4月なら、それ以降に起きた出来事や更新された仕様は知らない。
> **用語｜ハルシネーション：** 事実ではないことを自信を持って正しいかのように生成する現象。AIは「知らない」と言わず、もっともらしい嘘をつくことがある。

この特性を理解しておかないと、AIの回答を鵜呑みにして間違った情報を使ってしまう。AIの出力は「参考情報」として扱い、重要な判断は自分で確認する癖をつけることが大切。

## 1-2. AIにできること・できないこと

### エンジニアの業務でAIが得意なこと

| できること | 具体例 |
|---|---|
| コードの生成 | 「ボタンを押したらリストに追加するJSを書いて」 |
| コードの説明 | 「このコードが何をしているか説明して」 |
| バグの原因調査 | 「このエラーメッセージの原因を教えて」 |
| リファクタリング案の提示 | 「このコードをもっと読みやすくして」 |
| ドキュメントの下書き | 「この仕様をREADMEにまとめて」 |
| 調査・情報収集の補助 | 「〇〇の使い方を教えて」 |

### AIが苦手なこと・任せてはいけないこと

| できないこと・任せてはいけないこと | 理由 |
|---|---|
| 最新情報の提供 | 学習データに時差があるため |
| 社内固有のシステムや仕様の把握 | 学習データに含まれていないため |
| 生成したコードの動作保証 | AIは動くと思って出力するが、実際には動かないこともある |
| セキュリティ上の最終判断 | リスクの判断は人間が行う必要がある |
| 機密情報の取り扱い | 渡した情報がどう使われるかはポリシー次第（第2章で詳しく扱う） |

**AIを「賢いアシスタント」として使うのは正しい。でも「全部任せられる自律した存在」として扱うのは間違い。** 最終的な判断と責任は常に人間にある。

## 1-3. Gemini Enterprise とは何か

### Gemini の種類

Gemini には複数のバージョンがある。

| バージョン | 対象 | 特徴 |
|---|---|---|
| Gemini（無料） | 個人 | Google アカウントで誰でも使える。データが学習に使われる可能性がある |
| Gemini Advanced | 個人（有料） | より高性能なモデルが使える |
| **Gemini Enterprise** | **企業（Google Workspace）** | **企業契約。入力データは学習に使われない。管理者が利用状況を管理できる** |

このコンテンツで使うのは **Gemini Enterprise**。

### Gemini Enterprise を使う理由

無料版と最大の違いは**データの扱い**。

無料版では、入力した内容がGoogleのモデル改善のために利用される可能性がある。一方、Gemini Enterprise は企業契約のため、**入力データはGoogleのモデル学習には使われない**。

業務でAIを使う場合、コードや仕様など会社の情報を扱うことが多い。そのため、データポリシーが明確な Enterprise 版を使うことが前提になる。

> ただし「Enterprise だから何を入力しても安全」ではない。機密情報の取り扱いについては第2章で詳しく学ぶ。

### Gemini CLI とは

**Gemini CLI** は、ターミナル（コマンドライン）から Gemini を操作できる公式ツール（2025年5月・オープンソースで公開）。

ブラウザで画面を開かなくても、ターミナルから直接 Gemini に指示を出せる。さらに、テキストファイルや仕様書をそのまま渡して処理させたり、複数の操作を自動化したりできる。

このコンテンツでは、ブラウザの Web UI と Gemini CLI の両方を使いながら、用途に応じた使い分けを学んでいく。

## 1-4. なぜエンジニアがAIを使うのか

### 「書く時間」より「考える時間」を増やすため

コードを書く作業の中には、調べればわかること・決まったパターンがあること・繰り返しの作業が多く含まれている。

AIはこういった「すでに答えがある作業」を高速に処理することが得意。AIに任せることで、**「どう設計するか」「何を作るか」「どう判断するか」という人間が考えるべき部分に集中できる**。

### スピードの問題ではない

AIを使う理由を「速くなるから」と理解している人は多い。それは正しいが、半分しか正しくない。

重要なのは**「試行回数が増える」**こと。

コードの初稿をAIに出させて、それをレビューして修正する。このサイクルを繰り返すことで、1人で全部考えて書くより多くの選択肢を検討できる。「いいものを作る」ために使うのがAIの本質的な価値。

### 「使えた」は出発点にすぎない

AIツールは今も進化し続けている。今日の使い方が半年後には古くなることもある。「使えるようになった」という事実はすぐに陳腐化する。

重要なのは「使えた」ではなく、**変化するAIを自分で学び、使いこなし続けられるかどうか**だ。

新しいツールが出たとき自分でドキュメントを読んで試せるか。わからないことをAIに聞いて理解できるか。出力を批判的に評価できるか。この繰り返しが、エンジニアとしての成長をつくる。

### ② 「AIにできないこと」を自分でリストアップする

自分の今の業務や開発作業を思い浮かべながら、「これはAIに任せられないな」と思うことを3つ書き出してみよう。

書き出したら、Gemini にこう聞いてみる。

```
エンジニアの業務の中で、AIが苦手なこと・任せてはいけないことを教えてください。
```

自分のリストと回答を比べて、気づかなかった観点がないか確認する。

### ③ Gemini の回答の「限界」を体験する

以下のように聞いてみよう。

```
今日のニュースを教えてください。
```

次に、こう聞いてみる。

```

```

2つの回答から「AIが持っている情報の範囲」を体感する。

## 2-1. AIに渡した情報はどこへ行くのか

AIとの会話は「ただのチャット」ではない。入力した内容はサーバーに送信され、処理される。

多くのAIサービスでは、送信された内容が以下のいずれかの用途に使われる可能性がある。

| 用途 | 内容 |
|---|---|
| モデルの改善・学習 | 入力データをAIの性能向上に使用する |
| サービス品質の向上 | 不適切な出力のチェックなどに人間がレビューする |
| ログの保存 | 一定期間、会話内容がサーバーに保存される |

「AIに話しかけること」は、情報をインターネット上のサービスに送信することと同じ。この前提を忘れずに使うことが重要。

---

## 2-2. Gemini Enterprise と一般向け Gemini のデータポリシーの違い

Gemini には複数のバージョンがあり、データポリシーが異なる。

| バージョン | データの学習利用 | 管理者の利用管理 |
|---|---|---|
| Gemini（無料版） | 入力データが学習に使われる可能性がある | なし |
| Gemini Advanced（個人有料版） | 設定によって異なる | なし |
| **Gemini Enterprise（Google Workspace）** | **入力データはモデル学習に使われない** | **管理者が利用状況を管理可能** |

業務でAIを使う場合、Gemini Enterprise を使う最大の理由はここにある。**会社の情報がAIの学習データとして外部に漏れない**という保証がある。

ただし、これは「何でも入力していい」という意味ではない。

> Gemini Enterprise であっても、入力データはGoogle のサーバーで処理される。「学習に使われない」と「完全に安全」は別の話。機密情報の扱いは次のセクションで整理する。

---

## 2-3. AIに渡してはいけない情報

以下の情報は、たとえ Gemini Enterprise であっても AIに渡さない。

### 絶対に渡してはいけないもの

| 種類 | 具体例 |
|---|---|
| 認証情報 | APIキー・パスワード・トークン・秘密鍵 |
| 個人情報 | 氏名・住所・電話番号・メールアドレス・マイナンバー |
| 機密性の高い社内情報 | 未公開の財務情報・M&A情報・顧客の個人データ |
| 規制対象データ | 医療情報・金融情報など法的に保護されているデータ |

### 渡す前に考えるべきもの

| 種類 | 判断基準 |
|---|---|
| 社内のソースコード | 公開しても問題ないか・競合に見られて困る内容を含まないか |
| 顧客に関する情報 | 個人が特定できる情報が含まれていないか |
| 社内の仕様書・設計書 | 機密指定されていないか |

迷ったときの判断基準：**「このテキストをインターネットに公開しても問題ないか？」** と考える。問題があるなら渡さない。

---

## 2-4. コードをAIに渡す際のリスク

コードを渡してレビューや修正を頼むことは、AIの得意な使い方のひとつ。ただし、コードの中に機密情報が混入していることがある。

### よくある危険なパターン

```javascript
// 絶対にやってはいけない例
const API_KEY = "AIzaSyXXXXXXXXXXXXXXXXXXXXXXXX";
const DB_PASSWORD = "mypassword123";

fetch(`https://api.example.com/data?key=${API_KEY}`);
```

コード内に直接書かれた認証情報はそのままAIに送信される。

### 正しい扱い方

認証情報は `.env` ファイルに分離して管理し、コードには書かない。

> **用語｜`.env` ファイル：** APIキーやパスワードなど機密情報を記述する専用ファイル。コードとは別ファイルに分離することで、誤ってGitHubに公開するリスクを減らせる。

```javascript
// 正しい例：環境変数から読み込む
const API_KEY = process.env.API_KEY;
```

```bash
# .env ファイル（AIには渡さない・GitHubにもコミットしない）
API_KEY=AIzaSyXXXXXXXXXXXXXXXXXXXXXXXX
DB_PASSWORD=mypassword123
```

AIにコードを渡す前に、認証情報や個人情報が含まれていないかを必ず確認する。

## 2-5. GitHubにコミットしてはいけないもの

GitHubのパブリックリポジトリは世界中に公開される。プライベートリポジトリでも、誤って公開設定にしてしまうリスクや、アカウントが侵害されるリスクがある。

### コミットしてはいけないファイル・情報

| ファイル・情報 | 理由 |
|---|---|
| `.env` ファイル | APIキーやパスワードが含まれる |
| `*_secret.json` などの認証情報ファイル | 同上 |
| `node_modules/` | 容量が大きく不要・`package.json` から復元できる |
| ビルド成果物（`dist/`, `build/`） | ソースから生成できる |
| OS・エディタが生成するファイル | `.DS_Store`, `.vscode/settings.json` など |

### `.gitignore` で守る

`.gitignore` ファイルに記載されたファイルは Git の管理対象から除外される。プロジェクトを作ったら最初に設定する。

```bash
# .gitignore の基本的な記述例
.env
.env.local
node_modules/
dist/
.DS_Store
```

> 一度コミットしてGitHubにプッシュしてしまった認証情報は、**その瞬間に漏洩したとみなす**。履歴から削除しても、すでにスキャンされている可能性がある。漏洩したAPIキーは即座に無効化・再発行する。

## 2-6. AIが生成したコードの著作権・ライセンス問題

AIが生成したコードは学習データのパターンから生成される。そのため、学習データに含まれたオープンソースコードに似たコードが出力される場合がある。

現時点（2025年）では、AIが生成したコードの著作権については法的にまだ議論が続いている部分もある。ただし、以下の点は意識しておくべき。

- 商用プロジェクトで使う場合、生成されたコードが特定のライセンス（GPL など）に抵触しないか確認することが望ましい

> **用語｜GPL（GNU General Public License）：** オープンソースライセンスの一つ。GPLのコードを使ったソフトウェアはソースコードの公開義務が生じる場合がある。商用利用には注意が必要。
- 生成されたコードをそのままコピー&ペーストするだけでなく、**内容を理解した上で使う**ことが重要
- 「AIが生成したから責任はない」は通用しない。コードの品質と法的問題の責任は使った人間にある

## 2-7. 人間が管理すべき判断とAIに任せてよい判断の線引き

AIは強力なアシスタントだが、すべての判断を任せることはできない。

### AIに任せてよいこと

- コードの初稿を生成する
- 書いたコードの問題点を指摘させる
- ドキュメントの下書きを作る
- エラーメッセージの原因を調べる
- 複数の実装方法の選択肢を出させる

### 人間が必ず判断すること

| 判断 | 理由 |
|---|---|
| セキュリティに関わる設計 | AIは脆弱性を見落とすことがある |
| 本番環境へのデプロイ | 影響範囲が大きく取り返しがつかない |
| 顧客・ユーザーへの影響がある変更 | AIは文脈やビジネス上の影響を完全には理解できない |
| 生成されたコードが正しいかどうかの最終確認 | AIは間違いを自信満々に返すことがある |
| 何をAIに渡すかの判断 | このセクションで学んだこと |

**AIは「作業を速くするツール」であり、「責任を引き受けるツール」ではない。** 最終的な判断と責任は常に人間にある。

## 3-5. 基本的な Git 操作

Git の基本的な流れは「変更を記録してリモート（GitHub）に送る」。

> **用語｜ステージング：** コミット（記録）する前に「この変更を記録対象に含める」と選択する操作。`git add` で行う。全変更を一度にコミットするのではなく、関連する変更だけをまとめてコミットできる。

### 基本コマンド

```bash
# リポジトリを初期化する（新規作成時）
git init

# リモートリポジトリをローカルにコピーする
git clone リポジトリのURL

# 変更したファイルをステージングする（コミット対象に追加）
git add ファイル名
git add .   # カレントディレクトリ以下すべて

# ステージングされた変更をコミットする（記録する）
git commit -m "変更内容のメッセージ"

# リモートリポジトリに送る
git push origin main

# 現在の状態を確認する
git status

# コミット履歴を確認する
git log --oneline
```

### 実際の流れ

```bash
# 1. GitHubからリポジトリをクローンする
git clone git@github.com:自分のユーザー名/todo-app.git

# 2. ディレクトリに移動
cd todo-app

# 3. ファイルを作る
touch index.html

# 4. 変更をステージング
git add index.html

# 5. コミット
git commit -m "最初のコミット"

# 6. プッシュ
git push origin main
```

GitHubのページを更新して `index.html` が追加されていれば成功。

## 3-9. Web UI（Google Workspace）と CLI の使い分け

どちらを使うかは、何をしたいかで決める。

| 用途 | 向いているツール | 理由 |
|---|---|---|
| 気軽に質問する | Web UI | ブラウザで開くだけで使える |

どちらが優れているというものではない。状況に応じて使い分けることが重要。

## 4-1. 公式ドキュメントはなぜ読むべきか

### ブログ・Qiita・記事との違い

ネットで調べると、ブログや技術記事がたくさん出てくる。読みやすいし、すぐ使えそうな情報が多い。でも、これらには共通の問題がある。

**書かれた時点の情報しか保証されない。**

AIツールのバージョンが変わったり、APIの仕様が変わったりすると、記事の内容が古くなる。古い情報をそのまま試してエラーが出て詰まる、というのはよくある話。

一方、公式ドキュメントは開発元が管理しているため：

- **最新の仕様が書かれている**
- **動作が保証されている手順が書かれている**
- **変更があった場合に更新される**

公式ドキュメントを読む習慣がある人とない人では、詰まったときの解決速度が大きく変わる。

### 「難しそう」という先入観を捨てる

公式ドキュメントは英語が多い、難しい言葉が並んでいる、と感じる人は多い。でも全部読む必要はない。

**目的のページに直接飛んで、必要な部分だけ読む** で十分。わからない単語は Gemini に説明してもらえばいい。

## 4-3. Gemini 自身に「Gemini について」聞く

「AIでAIのことを調べる」は一見おかしく聞こえるが、実際にはとても有効な方法。

### なぜ有効なのか

Gemini は自分自身のドキュメントや仕様についての情報も学習している。難しいドキュメントを読む前に、まず Gemini に「わかりやすく説明して」と頼むことで、ドキュメントの理解が速くなる。

ただし注意点が1つ。

> **Gemini の回答には学習データのカットオフがある。** 最新の機能やアップデートについては、正確でない場合がある。Gemini の回答を「理解のための補助」として使い、最終確認は公式ドキュメントで行う。

### 具体的な使い方

ドキュメントを読んでいてわからない部分をコピーして、Gemini に貼り付ける。

[ドキュメントの文章をここに貼り付ける]
```

または、特定の機能について概要を聞く。

```
Gemini CLI の --context オプションは何をするものですか？
具体的な使用例も教えてください。
```

| 確認すること | 方法 |

| 記事の公開・更新日 | ページ上部か下部に表示されていることが多い |
| 対象バージョンの明記 | 「Gemini 1.5 Pro 時点」などの記述があるか |

| 確認すること | 判断基準 |

| 書いた人・組織 | 公式か、信頼できる個人や組織か |
| 参照元が明記されているか | 「公式ドキュメントによると」などの引用があるか |
| 実際に試した結果があるか | 「やってみたら動いた」という実証があるか |

### Gemini に情報の確認を頼む

```
以下の情報は現在でも正確ですか？また、この情報はいつ頃のものですか？

[確認したい情報を貼り付ける]
```

Gemini も間違えることがあるので、重要な情報は複数の情報源で確認する習慣をつける。

## 4-5. 「わからないこと」を言語化してAIに聞く技術

AIへの質問で最もよくある失敗は、「質問が曖昧すぎる」こと。

### 悪い質問と良い質問の違い

**悪い例：**
```
エラーが出ました。どうすればいいですか？
```

**良い例：**
```
以下のエラーが出ています。
環境：macOS、Node.js v20、Gemini CLI v1.0
実行したコマンド：gemini --context ./spec.md "HTMLを作ってください"
エラーメッセージ：[エラー内容をここに貼り付ける]

このエラーの原因と解決方法を教えてください。
```

### 質問に含めるべき情報

1. **何をしようとしていたか**（目的）
2. **何をしたか**（実行した操作・コマンド）
3. **何が起きたか**（エラーメッセージ・実際の結果）
4. **環境情報**（OS・ツールのバージョンなど）

### わからないことを言語化するコツ

「何がわからないかがわからない」状態のときは、まず以下を Gemini に聞く。

```
[学んでいる内容] について理解できているかを確認したいです。
私の理解が正しいか確認してください。

私の理解：[自分の理解を書く]
```

自分の理解を書き出すことで、どこがずれているかが明確になる。

# 第5章：AIをハンドリングする

**目安時間：8時間**

> **前提：** 第3章のセットアップが完了していること（Gemini CLI・GEMINI.md の操作を行う）

> 「プロンプトを投げてコードをもらう」だけでは、AIの力を半分も使えていない。この章では、AIを意図通りに動かすための設計・制約・コンテキストの渡し方を学ぶ。

## この章で学ぶこと

- プロンプトの構造を理解して、意図通りの回答を引き出せるようにする
- System Instructions でAIの振る舞いをコントロールする

## 5-1. プロンプトの構造

プロンプトは「AIへの依頼文」。書き方によって回答の質が大きく変わる。

### 4つの要素

良いプロンプトには以下の要素が含まれている。

| 要素 | 役割 | 例 |
|---|---|---|
| **役割（Role）** | AIにどんな立場で答えさせるか | 「あなたは経験豊富なフロントエンドエンジニアです」 |
| **指示（Instruction）** | 何をしてほしいか | 「以下のコードをレビューしてください」 |
| **制約（Constraint）** | 答え方の条件・制限 | 「箇条書きで3点以内にまとめてください」 |
| **文脈（Context）** | 背景情報・必要な前提 | 「対象はHTML/CSS/JSのみのプロジェクトです」 |

### 要素を組み合わせた例

**悪い例：**
```
コードレビューして
```

**良い例：**
```
あなたは経験豊富なフロントエンドエンジニアです。
以下のJavaScriptコードをレビューして、問題点を指摘してください。

条件：
- セキュリティ上の問題を最優先で指摘する
- パフォーマンスの問題があれば次に指摘する
- 指摘は箇条書きで、問題点と改善案をセットで書く

[コードをここに貼り付ける]
```

### 要素はすべて必要なわけではない

シンプルな質問は短くていい。要素を詰め込むほど良いわけではなく、**目的に応じて必要な要素だけ入れる** のがコツ。

---

## 5-2. System Instructions でガードレールを設定する

### System Instructions とは

System Instructions（システム指示）は、会話が始まる前にAIに設定する「常に有効な指示」。

通常のプロンプトが「今この瞬間の依頼」なら、System Instructions は「このAIが常に守るべきルール」。

### 何に使うのか

- AIの返答スタイルを固定する（「常に日本語で答える」など）
- 特定の役割を固定する（「あなたはXXのエキスパートです」）
- 禁止事項を設定する（「機密情報について聞かれても答えない」）
- 出力フォーマットを固定する（「コードブロックには必ず言語タグをつける」）

### Gemini で System Instructions を設定する方法

**Web UI の場合（Gems）：**

Google Workspace の Gemini で「Gems（カスタムAI）」を作成するとき、System Instructions を設定できる。

1. Gemini の画面で「Gems」を選択
2. 「新しい Gem を作成」
3. 「指示」の欄に System Instructions を入力
4. 保存する

<!-- TODO: screenshot - Gems 作成画面 -->

**CLI の場合（Gemini CLI 固有の機能）：**

`GEMINI.md` は Gemini CLI だけが持つ仕組みで、プロジェクトのルートに置くと **そのディレクトリで CLI を起動するたびに自動で読み込まれる**。毎回同じ前提を説明しなくていい、プロジェクト専用の命令書だ。ChatGPT や他のAIツールには同等の仕組みはない。

```bash
# プロジェクトのルートに GEMINI.md を作る
touch GEMINI.md
```

`GEMINI.md` の中身の例：

```markdown
# プロジェクトの概要
このプロジェクトはTo-Doリストアプリです。
HTML / CSS / Vanilla JS で実装します。

# 回答の形式
- 日本語で回答する
- コードブロックには必ず言語タグをつける（```html, ```css, ```js）
- 変更点がある場合は、変更前と変更後を明示する

# 制約
- フレームワーク（React, Vue など）は使わない
- 外部ライブラリは使わない
- コードには日本語のコメントを入れる
```

### System Instructions の設計のポイント

- **具体的に書く**：「わかりやすく答えて」より「箇条書きで3点以内にまとめて」の方が効果がある
- **矛盾を作らない**：指示同士が矛盾していると、AIが混乱した回答を返す
- **短く保つ**：長くなりすぎると守られない指示が出てくる

## 5-3. コンテキストの渡し方

AIは「与えられた情報」をもとに回答を作る。前提情報を渡さずに質問すると、的外れな回答が返ってくることがある。

### 何をコンテキストとして渡すか

| 渡すもの | 効果 |
|---|---|
| 仕様書・要件定義 | 何を作るかをAIが理解できる |
| 既存のコード | 「このコードに合わせて」という修正依頼ができる |
| エラーメッセージ | 原因の特定精度が上がる |
| プロジェクトの制約 | フレームワーク・バージョンなどの条件を踏まえた回答になる |
| これまでの会話内容 | 前提を共有したまま話を続けられる |

### コンテキストを渡すときの注意

- **第2章で学んだ通り、機密情報は渡さない**
- コンテキストが多すぎると重要な部分が埋もれる。渡す情報は絞る
- 「関係ありそうだから全部渡す」より「この質問に必要な部分だけ渡す」

---

## 5-4. CLI でコンテキストファイルを渡す

Gemini CLI では、ファイルの内容をそのままコンテキストとして渡せる。

### `--context` オプション

```bash
# spec.md をコンテキストとして渡す
gemini --context ./spec.md "この仕様書に基づいてHTMLの骨格を作ってください"
```

### パイプを使う

> **用語｜パイプ（`|`）：** あるコマンドの出力を、次のコマンドの入力として渡す仕組み。`cat file.md | gemini "..."` なら「file.mdの内容をgeminiに渡す」という意味になる。

```bash
# ファイルの内容をパイプで渡す
cat ./index.html | gemini "このHTMLのコードに問題はありますか？"
```

### 複数ファイルを渡す

```bash
# 複数ファイルを渡す場合
gemini --context ./spec.md --context ./index.html "仕様書に基づいて、このHTMLを修正してください"
```

### `GEMINI.md` との組み合わせ

`GEMINI.md` にプロジェクト全体の前提を書いておき、個別のファイルを `--context` で渡す。こうすることで、毎回同じ前提を説明しなくて済む。

```
GEMINI.md → プロジェクト全体の前提・ルール
--context → 今回の作業に必要なファイル
プロンプト → 今回やってほしいこと
```

## 5-5. AIの回答を引き出すコツ

### 一度で完璧な回答を求めない

AIとの対話は「一問一答」ではなく「キャッチボール」。最初から完璧な回答を期待するより、何回かやり取りして精度を上げていく方が結果がいい。

```
1回目：大まかな方向性を出させる
2回目：「〇〇の部分をもっと詳しく」「△△は不要」と修正
3回目：「他の選択肢はあるか」「このアプローチの問題点は」と深掘り
```

### 「なぜ？」を聞く

回答が返ってきたら、そのまま使うのではなく「なぜそうするのか」を聞く。

```
先ほどの回答で、[特定の部分] をそうした理由を教えてください。
他のアプローチと比べてどちらが良いですか？
```

理由を理解することで、生成されたコードを自分で判断できるようになる。

### 「制約を与えて絞る」か「制約をなくして広げる」か

| 目的 | やること |
|---|---|
| 特定の条件に合う答えが欲しい | 制約を明示する |
| 選択肢を出してほしい | 「3つの方法を提示してください」 |
| 自分の考えを確認したい | 「この方法の問題点を教えてください」 |
| 全体像をつかみたい | 「まず概要だけを説明してください」 |

## 5-6. AIが間違えやすいパターンを知る

AIは間違えることがある。よくある間違いのパターンを知っておくと、出力を見たときに「これは怪しい」と気づけるようになる。

### パターン① 自信満々に間違える（ハルシネーション）

AIは間違っていても断定的に答えることがある。「〇〇です」という表現を信用しすぎない。

**見分け方：** 重要な事実（関数名・APIの仕様・ライブラリのバージョンなど）は公式ドキュメントで確認する。

### パターン② 動かないコードを返す

コードが「見た目は正しそう」でも、実際に動かすとエラーが出ることがある。

**見分け方：** 生成されたコードは必ず自分で実行して確認する。動くまでは完成とみなさない。

### パターン③ 古い情報を使う

学習データのカットオフより後にアップデートされた情報は知らない。

**見分け方：** バージョンに依存する情報（APIの仕様・非推奨になった機能など）は必ずドキュメントで確認する。

### パターン④ 曖昧な指示を都合よく解釈する

指示が曖昧なとき、AIは「もっともらしい解釈」で答える。意図と違う方向に進むことがある。

**見分け方：** 回答が意図と違う場合は、指示の曖昧な部分を特定して言い直す。

### パターン⑤ 途中で仕様を忘れる

会話が長くなると、最初に伝えた前提を守れなくなることがある。

**見分け方：** 「最初に〇〇という条件を伝えましたが、それは守られていますか？」と確認する。

# 第6章：バイブコーディング入門

**目安時間：7時間**

> **前提：** 第5章を終えていること（プロンプト設計・GEMINI.md・コンテキストの渡し方を理解している）

> バイブコーディングとは、AIとの対話でコードを作り上げていく開発スタイル。「自分で全部考えて書く」ではなく「AIと一緒に作る」。ただし、出てきたコードを理解せずに使うのはリスクがある。この章ではコードを生成させながら、同時に読む・判断する力を身につける。

---

## この章で学ぶこと

- バイブコーディングの概念とサイクルを理解する
- 生成されたコードを読んで評価できるようになる
- AIがよくやる生成ミスのパターンを知る
- 間違いを見つけたとき、AIに修正させる方法を身につける

---

## 6-1. バイブコーディングとは何か

### 定義

バイブコーディング（Vibe Coding）は、AIとの対話を中心にコードを作っていく開発スタイル。

「自分でコードを書く」のではなく、「AIに作らせて、確認して、修正させる」というサイクルを繰り返す。

### 従来の開発との違い

| 従来の開発 | バイブコーディング |
|---|---|
| 仕様を理解 → 自分で設計 → コードを書く | 仕様をAIに伝える → AIが生成 → 確認・修正を繰り返す |
| コードを書く時間が長い | AIとの対話時間が長い |
| コードを書く力が必要 | コードを読む力・指示を出す力が必要 |
| 詰まると調査に時間がかかる | 詰まったらAIに聞ける |

### 「書けなくても開発できる」は半分正しい

バイブコーディングは「コードを書けなくてもアプリが作れる」という文脈で語られることがある。これは一面では正しいが、落とし穴がある。

**生成されたコードを読めないと：**
- バグがあっても気づかない
- セキュリティ上の問題があっても見逃す
- 意図と違うコードが動いていても判断できない

**バイブコーディングで本当に必要な力は「読む力」。**

AIにコードを書かせながら、「このコードは何をしているか」「意図通りか」を判断できる状態を目指す。

---

## 6-2. コード生成のサイクル

バイブコーディングの基本サイクルは以下の4ステップ。

```
① 依頼する
　　↓
② 出力を確認する
　　↓
③ 動かして確認する
　　↓
④ 問題があれば修正指示を出す → ① に戻る
```

### ① 依頼する

仕様や条件を整理してAIに渡す。5章で学んだプロンプトの構造を使う。

**ポイント：** 一度に大きな機能を依頼するより、小さく分けて依頼する方がうまくいく。

```
# 悪い例（一度に大きく頼みすぎ）
To-Doリストアプリを作ってください

# 良い例（小さく分けて頼む）
まず、テキスト入力欄と追加ボタンだけのHTMLを作ってください。
CSSとJSはまだ必要ありません。
```

### ② 出力を確認する

コードが返ってきたら、いきなりコピペせずに読む。

確認すること：
- 依頼した内容が実装されているか
- 余計なものが含まれていないか
- 知らない書き方が含まれていないか（わからない部分はAIに聞く）

### ③ 動かして確認する

コードをファイルに貼り付けてブラウザで確認する。見た目が合っているか、機能が動くかを確認する。

動かない・見た目が違う場合は次のステップへ。

### ④ 修正指示を出す

問題を具体的に伝えて修正させる。

```
# 曖昧な修正指示（悪い例）
なんか違います。直してください。

# 具体的な修正指示（良い例）
ボタンを押してもリストに追加されません。
JavaScriptのaddTask関数が呼ばれていないようです。
ボタンのonclick属性が正しく設定されているか確認して修正してください。
```

---

## 6-3. 生成されたコードを読む視点

生成されたコードを評価するために必要な視点を整理する。

### 視点① 動くかどうか

最初の確認はシンプルに「動くかどうか」。ブラウザで開いて、意図した通りに動作するか確認する。

### 視点② 意図通りかどうか

動いていても、依頼した内容と合っているかは別の話。

- 「ボタンを押したらリストに追加」という依頼に対して、ページをリロードして追加されていないか
- 削除ボタンが「削除」ではなく「完了」の動きをしていないか

### 視点③ 理解できるかどうか

コードの各行が「何をしているか」を説明できるか確認する。

説明できない部分があれば Gemini に聞く。

```
以下のコードについて、各行が何をしているか説明してください。

[コードを貼り付ける]
```

### 視点④ 問題がないかどうか

機能が動いていても、以下の問題が潜んでいることがある。

| 確認ポイント | 内容 |
|---|---|
| セキュリティ | ユーザーの入力をそのままHTMLに書き出していないか（XSS） |

> **用語｜XSS（クロスサイトスクリプティング）：** ユーザーの入力をそのままHTMLに埋め込むことで、悪意あるスクリプトが実行されてしまう脆弱性。`innerHTML` ではなく `textContent` を使うことで防げる。第7章で実際に確認する。
| 不必要な処理 | 使っていない変数・関数が含まれていないか |
| 依頼した制約が守られているか | 「外部ライブラリを使わない」など、System Instructions の制約を守っているか |

---

## 6-4. よくある生成ミスのパターンと見分け方

### ミス① イベントリスナーが重複する

「ボタンを押すたびに動作が増えていく」という症状が出たら、イベントリスナーが複数回登録されている可能性がある。

```javascript
// 問題のあるパターン
function setup() {
  const btn = document.getElementById('addBtn');
  btn.addEventListener('click', addTask);  // 関数が呼ばれるたびに追加される
}
```

### ミス② `null` チェックがない

要素が見つからない場合にエラーが起きる。

```javascript
// 問題のあるパターン
const input = document.getElementById('taskInput');
const value = input.value;  // input が null の場合にエラー

// 安全なパターン
const input = document.getElementById('taskInput');
if (!input) return;
const value = input.value;
```

### ミス③ HTMLのid名が一致しない

HTMLのidとJSで参照しているidが違う。コードを通しで見たときに気づきやすい。

```html
<!-- HTML -->
<input id="task-input" />
```

```javascript
// JS（idが違う）
const input = document.getElementById('taskInput');  // null になる
```

### ミス④ 古い書き方が混じる

学習データに古いコードが含まれているため、非推奨の書き方が出てくることがある。

```javascript
// 古い書き方
var x = 1;

// 現在の書き方
const x = 1;
let y = 2;
```

見つけたら修正を依頼する。

## 6-5. AIへのフィードバックの仕方

### 修正指示の基本

効果的な修正指示には以下の情報が含まれている。

1. **どこが問題か**（ファイル名・関数名・行番号など）
2. **どうなっているか**（現在の動作・エラーメッセージ）
3. **どうしてほしいか**（期待する動作）

```
addTask関数に問題があります。
現在の動作：ボタンを押すと入力値が追加されますが、入力欄がクリアされません。
期待する動作：タスク追加後に、入力欄を空にしてください。
```

### 部分的な修正を依頼する

コード全体を再生成させると、他の部分が変わってしまうことがある。修正したい部分だけを指定する。

```
以下のコードの addTask 関数のみを修正してください。
他の部分は変更しないでください。

[修正したい関数のコードを貼り付ける]
```

### 理由を聞きながら修正させる

修正後に「なぜそうしたか」を聞くことで、次に同じ問題が起きたときに自分で対処できるようになる。

```
修正してくれた内容について、なぜその方法を選んだか説明してください。
他にも修正方法はありますか？
```

## 7-7. 生成されたコードを通しでレビューする

### コード全体を確認する

すべての機能が実装できたら、コード全体を通してレビューする。

```bash
cat ./script.js | gemini "このJavaScriptコードをレビューしてください。
以下の観点で確認してください：
1. セキュリティ上の問題（XSSなど）
2. バグや意図しない動作の可能性
3. 不要なコードや重複した処理"
```

### リスク観点でのチェック（第2章の復習）

- 機密情報が含まれていないか
- ユーザーの入力をそのままHTMLに書き出していないか（XSS）

#### XSSの例

```javascript
// 危険な書き方（ユーザー入力をそのまま innerHTML に入れる）
listItem.innerHTML = input.value;

// 安全な書き方（textContent を使う）
listItem.textContent = input.value;
```

生成されたコードで `textContent` と `innerHTML` のどちらが使われているかを確認したら、必ずGeminiに理由を聞く。

```
生成されたコードで textContent を使っていますが、innerHTML ではなく textContent を使う理由を教えてください。
セキュリティの観点からも説明してください。
```

「動いたから OK」ではなく、**なぜその書き方が選ばれたか**を理解することが、第2章で学んだリスク管理の実践になる。

レビュー結果で問題が出たら、修正指示を出して直す。

## 7-8. コミット前チェック

コードをプッシュする前に必ず確認する。

### チェックリスト

```
□ .env などの機密ファイルが .gitignore に含まれているか
□ コード内にAPIキー・パスワードなどが直書きされていないか
□ git status で意図しないファイルがステージングされていないか
□ ブラウザで全機能が動くことを確認したか
```

### コミットメッセージのルール

後から見てわかるメッセージを書く。

```bash
# 悪い例

```

気に入ったメッセージが返ってきたらそのまま使う。微妙だったら「もっと具体的に」「動詞から始めて」と追加で指示して調整できる。

## 7-9. LocalStorage でデータを保存する

現状のアプリはページをリロードするとタスクが消える。LocalStorage を使うことで、ブラウザにデータを保存してリロード後も残るようにできる。

### LocalStorage とは

ブラウザが持つ簡易的なデータ保存の仕組み。サーバーなしで、ユーザーのブラウザにデータを永続的に保存できる。

### spec.md に追記する

まず仕様書を更新してから実装依頼する。

```markdown
## 追加機能
- タスクの追加・完了・削除のたびに LocalStorage に保存する
- ページを読み込んだとき、LocalStorage からタスクを復元して表示する
```

### Gemini に実装させる

```bash
gemini -p "spec.md の追加機能を実装してください。既存のコードは変更せず、LocalStorage の読み書き処理だけ追加してください。"
```

### 生成されたコードで確認すること

LocalStorage の操作には以下の3つが使われているはずなので、それぞれ何をしているか Gemini に聞いて確認する。

```javascript
localStorage.setItem('key', value)  // 保存
localStorage.getItem('key')         // 読み込み
localStorage.removeItem('key')      // 削除
```

確認できたらブラウザでリロードしてタスクが残ることを検証し、コミットする。

```bash
git add script.js
git commit -m "LocalStorageでタスクを永続化"
```
